  day/time + weather system was explicitly deferred to Phase 4 in the P3 HLD's
  non-goals).
- **Phase 5 — Reach:** touch/mobile input, accessibility, performance pass.

**Proportionality decision (SCALE directive + Phase-3 precedent).** Both phases
are predominantly presentation/content/platform work with **low design-risk** and
mostly-untestable output (Three.js geometry, DOM, device polling). The Phase-3
journal already established the right shape for this: **author the HLD directly,
carve out the one or two genuinely *pure & testable* seams, self-sign-off, and put
the effort into implementation + build/screenshot verification + review.** The
heavy 3-draft/judge/Codex design cycle that suited Phases 1-2's load-bearing
invariants (the shared penalty latch, the signalling machine) is disproportionate
here. So:
- **VALIDATE/DRIFT:** I authored both HLDs against **current HEAD** (files read
  directly this session: `main.ts`, `route.ts`, `scene.ts`, `cab.ts`, `train.ts`,
  `package.json`). Drift is **nil by construction** — the design targets the tree
  as it actually is (v0.4.0, 111 tests, render seam = `RenderView`, `mu` hardwired
  `ADHESION.wetNight`, no `src/input/` yet). No separate drift workflow: re-reading
  my own just-read assumptions is exactly the "width has diminishing returns" the
  directive warns against. Acceptance criteria are extracted into each HLD §4.
- **IMPLEMENT:** Workflow pipeline, one implementer per work item, **pure tested
  seams partitioned by file from impure adapters**. Impure Three.js/DOM that needs
  a screenshot loop an agent can't run: the agent produces the first cut (DoD =
  tsc/build clean + integrates), then I (lead) own the build+screenshot verify and
  fix any rendering bug in a follow-up commit — exactly how Phase 3's WI-2 ran
  (agent draft → lead `−Z` cab fix).
- **REVIEW:** Workflow, small multi-dimension panel (correctness / tests / fidelity
  to HLD / security-lite / style) with 2-of-3 adversarial verification of each
  finding; loop until a clean round.

**The load-bearing testable seams I carved (so these aren't pure-aesthetic phases):**
- **Phase 4:** weather is tied to **adhesion μ** *and* visuals via a pure
  `environmentParams(env)` — wet conditions lower μ (feeds the real physics), night
  darkens lighting, rain drives particle intensity. Default (night+rain) pins
  μ = 0.20 to **preserve Phase 0-3 physics exactly** (differential/no-regression).
- **Phase 5:** a device-agnostic **input model** — `intentFromActions` /
  `mergeActions` — so keyboard, gamepad and touch all map through one pure, tested
  function to `ControlIntent`; plus pure `accessibilityParams` (reduced-motion
  gate) and `qualityFor` (perf scaling).
