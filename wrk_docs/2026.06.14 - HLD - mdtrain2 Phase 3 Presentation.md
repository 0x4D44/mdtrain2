# HLD — mdtrain2 Phase 3: Presentation (wet-night cab + GTO audio)

**Status:** SIGNED-OFF · Date 2026-06-14 · **Approved by:** self-signed under delegated authority (Arthur granted "full autonomy, both")
**Provenance:** lead-authored directly (proportionality — Phase 3 is aesthetic/presentation with low design-risk and mostly-untestable output; the heavy ultracode design cycle that suited Phases 1-2's testable invariants is disproportionate here). Target `C:\language\mdtrain2` (Phase 0-2 merged). `package.json` 0.3.0 → 0.4.0.

This phase turns the placeholder night scene into the signature experience: a **GTO-inverter EMU driven first-person from a detailed cab, at night in the rain** — wipers, signal glow, wet rails — with the inverter "gearbox" whine rising under power. It binds entirely to existing sim state (`SimState`, `ControlState`, `SafetyState`, `AwsState`, the route signals/aspects); it adds **no** new gameplay and changes **no** sim-core logic.

---

## 1. Scope & non-goals

### In scope
- **3D cab interior** (`src/render/cab.ts`): driver's-eye cab — windscreen frame + a sweeping **wiper**, desk, a **power controller** lever and a **brake** lever that move with `ControlState` (notch/brakeStep), a **reverser**, a circular **speedometer** (dial + needle from speed), the **AWS sunflower** disc (black ↔ black/yellow from `AwsState.sunflower`), and **DRA / DSD / penalty** indicator lamps. Parented to the camera (moves with the view).
- **Wet-night world** (expand `src/render/scene.ts`): night lighting (dim hemisphere + a cool moon directional + fog already present, deepened), **rain** (instanced streak particles + animation), **3D signal heads** at each `route.signals` starter that **glow** with the live aspect colour (RED/YELLOW/DOUBLE_YELLOW/GREEN derived from `aspectAt` over the served set), and **wet-rail** specular sheen (reflective rail material + a faint signal-colour glow on the railhead near a lit signal).
- **GTO inverter audio** (`src/audio/`): a procedural WebAudio engine producing the rising "gearbox" whine (stepped/ramping oscillator tones) gated by traction, plus speed-scaled rolling/rail noise and a brake hiss. The **sim-state → audio-param mapping is a pure, tested function**; the WebAudio graph is a thin impure adapter.
- A pure **HUD coexists** (keep the DOM HUD from Phase 1-2 for readable text: speed, limit, next stop, aspect, lamps).

### Non-goals
- Photoreal graphics; full lineside scenery / buildings / stations modelling (the world stays schematic); reflections via render-to-texture (use cheap material sheen, not a mirror pass).
- Music; voice/PA. Day/weather selector (wet-night is the fixed signature; a time-of-day system is Phase 4).
- Gamepad/touch (Phase 5). No sim-core/gameplay changes; no new env vars / feature flags.

## 2. The design

### 2.1 The render seam (how it binds)
Today `scene.render(chainage)` takes one scalar. Phase 3 widens the per-frame contract to a **render view** projected from sim state in `main.ts`:

```ts
// what main.ts assembles each frame and passes down
interface RenderView {
  chainage: number; speed: number; dt: number;     // camera + gauges + rain/wiper animation
  controls: ControlState; safety: SafetyState; aws: AwsState;  // cab gauges/levers/lamps
  served: ReadonlySet<string>;                      // for live signal aspects (via aspectAt)
}
```
- `createScene(parent, route)` → `{ render(view: RenderView), resize() }` (signature widened; the camera still follows `view.chainage`).
- `createCab(camera)` → `{ update(view: CabView) }` where `CabView` is the small projection the cab needs (notch index, brakeStep, reverser, speedMph, sunflower, dra/dsd/penalty booleans, wiperPhase). The cab meshes are children of the camera so they render as the foreground.
- The HUD (`hud.ts`) is unchanged.

### 2.2 The cab (`src/render/cab.ts`) — impure, screenshot-verified
Built once from primitive geometry (boxes/cylinders/extrusions; no external assets): a dark windscreen surround, a desk slab, two upright levers (power, brake), a small reverser, a round speedo face with a needle `Mesh` rotated by speed, a flat sunflower disc whose material swaps black ↔ black-with-yellow-sector, three small emissive lamp discs. `update(view)` each frame: rotate the levers to angles derived from `notch`/`brakeStep`, rotate the speedo needle, set the sunflower/lamp materials, advance the **wiper** arm by a sine of `wiperPhase` (driven off accumulated `dt`, only when rain is on — i.e. always, this phase). Pure geometry, no logic worth unit-testing → its correctness is "renders + screenshot."

### 2.3 The wet-night world (expand `src/render/scene.ts`) — impure
- **Lighting:** keep the night background + fog; dim the hemisphere, add a low cool directional "moon"; everything reads dark so the signal glows and rain catch the eye.
- **Signals:** for each `route.signals[i]`, place a signal-head mesh at its `chainage` beside the track; each frame set the head's lit-lamp **emissive colour** from `aspectAt(route, i, view.served)` (RED/amber/green; DOUBLE_YELLOW = two amber). A small additive sprite/halo gives the wet "glow." Cheap; no shadow pass.
- **Rain:** one `THREE.Points` (or instanced thin streaks) of a few thousand particles in a box around the camera, scrolled downward + along with motion each frame off `dt`/`speed`, wrapped to stay around the camera (no per-particle allocation per frame).
- **Wet rails:** raise the rail material's metalness/lower roughness for a specular sheen; add a faint additive streak on the railhead under a lit signal (optional, cheap).

### 2.4 GTO audio
**`src/audio/params.ts` (NEW, PURE, TESTED)** — the only logic worth pinning:
```ts
interface AudioParams {
  whineHz: number;       // GTO inverter tone, rises with speed
  tractionGain: number;  // 0..1, whine audibility (driven by notch under power)
  rollGain: number;      // 0..1, rolling/rail noise, rises with speed
  brakeHissGain: number; // 0..1, follows brake demand
}
function audioParams(speedMps: number, notch: number, brake: number): AudioParams;
```
Shape: `whineHz` increases monotonically with `|speed|` (a base + scaled, optionally with the characteristic GTO step-ramp segments); `tractionGain` tracks `notch` (and eases to 0 when coasting); `rollGain` rises with `|speed|` (0 at rest); `brakeHissGain` tracks `brake` and `|speed|`. At rest with no power/brake → all gains ≈ 0 (silence).

**`src/audio/engine.ts` (NEW, impure)** — `createAudioEngine()` → `{ start(), update(p: AudioParams) }`: one `AudioContext`, a couple of oscillators (saw/triangle) for the whine through a gain, a filtered noise buffer for rolling, a noise burst/gain for brake hiss, master gain. `start()` is called on first user gesture (the existing pointer/key handler). `update` sets oscillator frequency and gains via `setTargetAtTime` for smoothness. No logic beyond wiring the params to nodes.

`main.ts` each frame: `audio.update(audioParams(state.speed, controls.powerNotch/POWER_NOTCHES, resolvedBrake))`.

### 2.5 What stays pure / what's untested
- **Pure + tested:** `audioParams` (and any trivial projection helper). 
- **Untested adapters (named):** `cab.ts`, the `scene.ts` world additions, `audio/engine.ts`, and the `main.ts` view assembly — Three.js/WebAudio, verified by build + headless screenshot + "AudioContext instantiates," exactly as Phase 1's render thread was. No new logic lives here.

## 4. Acceptance criteria (definition of done)

- **AUD1** `audioParams`: `whineHz` strictly increases with `|speed|` (e.g. 0 < 10 < 25 m/s).
- **AUD2** `tractionGain` increases with `notch` (0 at notch 0; higher at full).
- **AUD3** `rollGain` increases with `|speed|` and is ≈0 at rest.
- **AUD4** `brakeHissGain` tracks `brake` (0 at brake 0).
- **AUD5** at rest with no power and no brake, all gains ≈ 0 (silence) — and outputs are finite (no NaN) across the speed/notch/brake domain.
- **REG** the existing 105 sim tests still pass; `tsc` zero warnings.
- **BUILD** `npm run build` clean.
- **SMOKE** (headless) the app loads with **zero console/page errors**; a screenshot shows the **cab foreground** (windscreen/desk/levers/speedo) over a **dark rainy scene with at least one glowing signal**; the `AudioContext` is created without throwing (suspended is fine in headless).
- **NO-REGRESSION** controls + signalling still drive (HUD aspect updates; train moves on power); the cab levers/speedo/sunflower visibly track state in the screenshot.

## 5. Residual risks
- **Aesthetic quality is not autonomously verifiable** — "renders without error + plausible screenshot" ≠ "looks/sounds good." Flagged for human review (morning report). The GTO whine character especially wants an ear.
- **Performance:** rain particles + emissive signals + cab geometry must stay light (instancing, no per-frame allocation); verify the build runs at interactive frame rate (sanity, not a hard oracle).
- **Headless audio:** AudioContext may start suspended under automation; the smoke only asserts it instantiates, not that sound plays.
- **Cab parenting:** cab meshes as camera children must not z-fight the world / clip; tune near-plane and cab scale.

## 6. Decision log
- **D1** Bind to existing sim state via a per-frame `RenderView`/`CabView`/`AudioParams`; **no sim-core change**, no new gameplay.
- **D2** Author this HLD directly + self-sign-off (proportionality: aesthetic phase, low design-risk, mostly untestable) rather than the full ultracode design cycle. Effort goes to implementation + screenshot/build verification.
- **D3** Carve the one testable seam — `audioParams` — into a pure module with oracles; keep Three.js/WebAudio as thin named-untested adapters (mirrors the Phase-1 render-thread split).
- **D4** Cheap wet-night look (emissive glow + material sheen + particle rain), **not** render-to-texture reflections — fits the "schematic world" scope and the perf budget.
- **D5** Keep the DOM HUD; the 3D cab gauges are additive immersion, not a replacement (readable text + immersive cab).
