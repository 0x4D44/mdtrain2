# HLD — mdtrain2 Phase 5: Reach (touch / gamepad, accessibility, performance)

**Status:** SIGNED-OFF · Date 2026-06-15 · **Approved by:** self-signed under delegated authority (Arthur granted "design+implement in one run" + "full autonomy, both")
**Provenance:** lead-authored against current HEAD (Phase 0-4 merged, v0.5.0). The load-bearing seam — a device-agnostic input model — is carved pure & tested; touch DOM / gamepad polling are thin screenshot-verified adapters. `package.json` 0.5.0 → 0.6.0.

Phase 5 widens *reach*: the sim becomes drivable by **touch** (on a phone/tablet) and **gamepad** as well as the keyboard, **respects `prefers-reduced-motion`**, and gains a small **performance** tier so it stays smooth on weak GPUs. The unifying idea: every device produces the same abstract **`InputActions`**, merged and mapped to the existing `ControlIntent` by **one pure, tested function** — so keyboard parity is provable and new devices are just new producers.

---

## 1. Scope & non-goals

### In scope
- **Device-agnostic input model** (`src/input/intent.ts`, NEW, PURE, TESTED): an
  `InputActions` record (the same ten control actions as today + `cycleEnvironment`
  + `anyActivity`), with `NO_ACTIONS`, `mergeActions(...sources)` (OR the fields),
  `intentFromActions(actions)` → `ControlIntent`, and pure producers
  `keyboardActions(edgeSet)` and `gamepadActions(buttons, axisX)`. The current inline
  `intentFromKeys` is *replaced* by `keyboardActions` + `intentFromActions` with
  **provable parity** (acceptance REG-INPUT).
- **Gamepad** (`src/input/gamepad.ts`, NEW, impure shell): poll
  `navigator.getGamepads()`, edge-detect buttons (so a held button fires one detent,
  matching the keyboard's no-auto-repeat rule), produce `InputActions` via the pure
  `gamepadActions`.
- **Touch UI** (`src/input/touch.ts` + DOM/CSS, NEW, impure): an on-screen control
  pad (power +/−, brake +/−, reverser F/N/R, DRA, acknowledge, emergency, env-cycle)
  shown on coarse-pointer / narrow viewports. Each tap is one edge → `InputActions`.
  Buttons carry ARIA labels. Screenshot-verified on a narrow viewport.
- **Accessibility** (`src/ui/motion.ts`, NEW, PURE, TESTED): `motionParams(prefersReducedMotion)`
  → `{ rainScale, wiperEnabled }`; when reduced motion is requested, rain is cut and
  the wiper parked. `main.ts` reads `matchMedia('(prefers-reduced-motion: reduce)')`.
  Plus ARIA labels on touch controls and a `lang`/viewport sanity pass.
- **Performance tier** (`src/render/quality.ts`, NEW, PURE, TESTED):
  `qualityFor({ coarsePointer, maxDevicePixelRatio, reducedMotion })` →
  `{ pixelRatioCap, rainCount }`. `createScene` takes the rain count + pixel-ratio cap
  from this (instead of the hard-coded 2400 / cap 2), so weak/mobile devices draw
  fewer particles at a capped ratio.

### Non-goals
- No new driving model / signalling / gameplay; no sim-core change (`src/sim/physics,
  simulation, controls, aws, route, environment, train` untouched except *imports*).
- No haptics, no on-screen joystick analog notching (buttons step detents, like keys),
  no full settings menu, no internationalisation beyond `lang`/labels.
- No new runtime dependency. No service worker / installable PWA (could be later).
- Not a visual redesign — the 3D world & HUD are unchanged except for the reduced-motion
  gate and the additive touch overlay.

## 2. The design

### 2.1 `src/input/intent.ts` (NEW, PURE, TESTED) — the seam
```ts
export interface InputActions {
  powerUp: boolean; powerDown: boolean;
  brakeUp: boolean; brakeDown: boolean;
  emergency: boolean;
  reverserFwd: boolean; reverserOff: boolean; reverserRev: boolean;
  toggleDra: boolean; acknowledge: boolean;
  cycleEnvironment: boolean;   // demo env cycle (was main's KeyE) — NOT a ControlIntent
  anyActivity: boolean;        // drives vigilancePing
}
export const NO_ACTIONS: InputActions;                      // all false
export function mergeActions(...sources: InputActions[]): InputActions; // OR each field; anyActivity = any
export function intentFromActions(a: InputActions): ControlIntent;      // → the existing intent shape
export function keyboardActions(edges: ReadonlySet<string>): InputActions; // today's keymap, now → actions
export function gamepadActions(buttons: readonly boolean[], axisX: number): InputActions; // pure pad map
```
- `intentFromActions` fills the existing `ControlIntent` (the ten controls +
  `vigilancePing = a.anyActivity`); `cycleEnvironment` is consumed separately by
  `main.ts` (it drives no train control).
- `keyboardActions` reproduces today's `intentFromKeys` exactly (W/S power, D/A brake,
  Backquote emergency, F/N/R reverser, L DRA, Q ack, **E cycleEnvironment**,
  `anyActivity = edges.size > 0`). Parity is an acceptance test.
- `mergeActions` OR-combines any number of producers so keyboard + gamepad + touch
  can all be live at once; `anyActivity` is the OR of every source's activity/any set field.
- `gamepadActions`: a fixed standard-pad map (e.g. right-cluster buttons = power up/down,
  triggers/shoulder = brake up/down, face button = acknowledge, dpad = reverser,
  `|axisX|`/buttons for env/DRA), `anyActivity` = any button or axis deflection.

### 2.2 `src/input/keyboard.ts` (NEW, thin impure) — edge ownership
Encapsulates the window keydown/keyup/blur listeners and the per-frame edge `Set`
that lives in `main.ts` today (keydown ignores `repeat`; keyup/blur prune; the frame
drains + clears). Exposes `edges(): ReadonlySet<string>` and `clear()`. Pure mapping
stays in `keyboardActions`. (Mechanical extraction — no behaviour change.)

### 2.3 `src/input/gamepad.ts` (NEW, impure shell)
`createGamepadSource()` → `{ actions(): InputActions }`. On each call: read
`navigator.getGamepads()[0]`, build a `boolean[]` of *edge-triggered* button presses
(compare to last frame so a hold = one detent) plus the raw `axisX`, and return
`gamepadActions(...)`. No-op safe when no pad is present (returns `NO_ACTIONS`).

### 2.4 `src/input/touch.ts` (+ DOM/CSS) (NEW, impure)
`createTouchControls(parent)` → `{ actions(): InputActions }`. Builds a fixed overlay
of labelled buttons (pointer/touch events set a one-frame edge per tap, mirroring the
keyboard's discrete-detent rule). Visible only when `matchMedia('(pointer: coarse)')`
or a narrow width (CSS media query); hidden on desktop so the desktop screenshot is
unchanged. Buttons have `aria-label`s and are real `<button>`s (focusable). `actions()`
returns the accumulated edges and clears them.

### 2.5 `src/ui/motion.ts` (NEW, PURE, TESTED) — reduced motion
```ts
export interface MotionParams { rainScale: number; wiperEnabled: boolean; }
export function motionParams(prefersReducedMotion: boolean): MotionParams;
```
`reduce` ⇒ `{ rainScale: 0, wiperEnabled: false }`; otherwise `{ rainScale: 1,
wiperEnabled: true }`. `main.ts` multiplies the env's `rainIntensity` by `rainScale`
and ANDs `wiperOn` with `wiperEnabled` before they reach the scene/cab.

### 2.6 `src/render/quality.ts` (NEW, PURE, TESTED) — perf tier
```ts
export interface QualitySettings { pixelRatioCap: number; rainCount: number; }
export function qualityFor(env: { coarsePointer: boolean; maxDevicePixelRatio: number; reducedMotion: boolean }): QualitySettings;
```
Desktop ⇒ `{ pixelRatioCap: 2, rainCount: 2400 }` (today's values, no regression).
Coarse-pointer (mobile) ⇒ lower cap (e.g. 1.5) and `rainCount ≈ 900`. `reducedMotion`
⇒ `rainCount: 0`. Always finite, `pixelRatioCap ∈ [1, 2]`, `rainCount ≥ 0`.

### 2.7 `src/render/scene.ts` (CHG, impure)
`createScene(parent, route, opts?: { rainCount?: number; pixelRatioCap?: number })`:
use the supplied rain count (default 2400) and pixel-ratio cap (default 2) instead of
the constants. No other behavioural change. (Rain already no-ops when `rainIntensity`
is 0, so reduced-motion → rainScale 0 stops it; a 0 `rainCount` also builds an empty
buffer safely.)

### 2.8 `src/main.ts` (CHG, impure) — compose the producers
Replace the inline `intentFromKeys`/edge plumbing with:
```ts
const kb = createKeyboardSource(); const pad = createGamepadSource(); const touch = createTouchControls(app);
const quality = qualityFor({ coarsePointer, maxDevicePixelRatio: devicePixelRatio, reducedMotion });
const scene = createScene(app, route, quality);
// per frame:
const actions = mergeActions(keyboardActions(kb.edges()), pad.actions(), touch.actions());
const intent = intentFromActions(actions);
if (actions.cycleEnvironment) env = cycleEnvironment(env);
const motion = motionParams(reducedMotion);
const ep = environmentParams(env);           // then rainIntensity *= motion.rainScale; wiperOn &&= motion.wiperEnabled
... reduceControls(controls, intent, state, safety) ...
kb.clear();
```
Still arithmetic-free: all decisions live in the pure functions. `reducedMotion`/
`coarsePointer` read once from `matchMedia`.

### 2.9 What stays pure / untested
- **Pure + tested:** `input/intent.ts`, `ui/motion.ts`, `render/quality.ts`.
- **Untested impure adapters (named):** `input/keyboard.ts`, `input/gamepad.ts`,
  `input/touch.ts`, the `scene.ts` opts plumbing, the `main.ts` composition — DOM/
  Gamepad/Three.js, verified by build + headless screenshots (desktop unchanged;
  narrow viewport shows touch controls; a tap moves a lever).

## 4. Acceptance criteria (definition of done)

- **IN1** `keyboardActions` maps each code to the correct action (W→powerUp, S→powerDown,
  D→brakeUp, A→brakeDown, Backquote→emergency, F/N/R→reverser*, L→toggleDra, Q→acknowledge,
  E→cycleEnvironment); `anyActivity === (edges.size > 0)`.
- **IN2** `gamepadActions` maps its fixed button/axis layout to the same action fields;
  no buttons + centred axis ⇒ `NO_ACTIONS` (anyActivity false).
- **IN3** `mergeActions` ORs every field across sources; `anyActivity` is true iff any
  source is active; `mergeActions()` with no args === `NO_ACTIONS`.
- **IN4** `intentFromActions(NO_ACTIONS)` is the all-false `ControlIntent` (incl.
  `vigilancePing === false`); a set action lights exactly its intent field;
  `vigilancePing === anyActivity`.
- **IN5** `cycleEnvironment` is carried on `InputActions` (not in `ControlIntent`) and a
  keyboard `E` and a touch env-button produce the same action.
- **REG-INPUT** for every subset of the mapped key codes,
  `intentFromActions(keyboardActions(edges))` deep-equals the **legacy** `intentFromKeys(edges)`
  (the Phase-1..4 keymap) — provable parity, no behavioural regression.
- **ACC** `motionParams(true) === { rainScale: 0, wiperEnabled: false }`;
  `motionParams(false) === { rainScale: 1, wiperEnabled: true }`.
- **QUAL** `qualityFor` desktop ⇒ `{ pixelRatioCap: 2, rainCount: 2400 }`; coarse-pointer ⇒
  lower cap and fewer particles; `reducedMotion` ⇒ `rainCount: 0`; all finite,
  `pixelRatioCap ∈ [1,2]`, `rainCount ≥ 0`.
- **REG** the existing **121** tests still pass; `tsc` zero warnings.
- **BUILD** `npm run build` clean.
- **SMOKE** (headless, zero console/page errors): (a) a **desktop** (wide) screenshot is
  unchanged — no touch overlay, keyboard still drives (train moves on power); (b) a
  **narrow-viewport** (e.g. 420×780) screenshot shows the **touch control overlay**; (c)
  tapping the touch **power** button raises the power notch (HUD `POWER` increases) — touch
  drives with no keyboard.

## 5. Residual risks
- **Aesthetic/ergonomic feel of the touch pad** (button size/placement on a real phone)
  needs human hands — flagged for review; headless only proves it renders + drives.
- **Gamepad** can't be exercised headlessly (no synthetic pad); covered by the pure
  `gamepadActions` oracle + a no-op-safe shell, not an end-to-end screenshot.
- **Reduced-motion media query** under automation may report `no-preference`; the pure
  `motionParams`/`qualityFor` are unit-tested directly rather than via the media query.
- **Parity risk:** the keyboard refactor must not change behaviour — REG-INPUT pins it
  against the retained legacy `intentFromKeys` (kept as a test reference).

## 6. Decision log
- **D1** One pure `InputActions` → `ControlIntent` map; every device is just a producer.
  Keyboard parity is provable (REG-INPUT) so the refactor is safe.
- **D2** Buttons (touch/gamepad) step detents like keys (one edge per press), preserving
  the deliberate no-auto-repeat rule; no analog notching.
- **D3** Accessibility + performance are *pure* params (`motionParams`, `qualityFor`)
  applied at the impure edges — testable, and they default to today's behaviour so
  desktop is a no-regression.
- **D4** Author HLD directly + self-sign-off (proportionality; presentation/platform phase).
- **D5** No PWA/service-worker/haptics/settings-menu — keep the phase tight and shippable.
