# HLD -- mdtrain2 Phase 2: Signalling

Status: under review (V4) · Date 2026-06-14 · Target `C:\language\mdtrain2` (Phase 0+1 merged, verified at HEAD) · `package.json` 0.2.0 → 0.3.0

This design adds a believable signalling / AWS / TPWS / SPAD layer to WESTFORD→EASTBANK by **growing state behind `tickAws`** and making **one scoped edit to `tickSafety`**: the carry-forward seed at `controls.ts:258` is narrowed to **DSD-only** (P2-RA), so AWS/TPWS reasons are re-derived fresh from the fold every tick and become releasable. The release machine (`tickSafety` `:277-281`), `resolveInputs`, and `reduceControls` are **byte-for-byte unchanged**. It is written testability-first: §2.0 derives the function shapes from the oracle list (§4), so the tested surface is minimal and traceable; the untested surface is two DOM lamps plus the `prevChainage`/`dir` plumbing in `main.ts`.

**The signal model (V4 — STARTERS ONLY).** The route's signals are **only the station STARTERS**. Each *intermediate* served station carries a **STARTER signal positioned just past its stop board** (the departure end), held **RED until the station is served** — the driver stops within the platform at a stand, which marks the station served and clears that starter. There are **no plain block signals** and there is **no terminus starter** (V4, R3-1/R3-2 below): with block signals removed, `aspectAt` is a clean **adjacent-starter cascade** (one R→Y→YY→G rung per starter), and the AWS sunflower is reset to BLACK on **serving a station** (a forward, reachable event) rather than on crossing a green-block magnet. The normal cycle is: *approach → AWS warning for the red starter ahead → ack → stop at the platform & serve → starter clears + sunflower resets to BLACK → depart across the now-GREEN post.* A SPAD / TPWS-TSS fires **only** if the driver passes the red starter **without having served** (a blow-through). A normal served stop is SPAD-free **by construction**, because the starter post sits beyond the platform half-length (R3-4, F4: `board + platformHalf < post − FRONT_OFFSET` for all three starters), so a legal stop anywhere within `±platformHalf` of the stop board serves the station (clearing its starter to GREEN) *before* the front ever reaches the RED post.

## 1. Scope and non-goals

### In scope
- **Route data**: fixed STARTER signal positions on `Route`, authored on WESTFORD_EASTBANK as **point features** (one chainage per signal post; post = `station.chainage + STARTER_OFFSET`; AWS-magnet and OSS-loop chainages are *derived* off the post chainage, not stored). **Exactly three starters** (Riverside, City Centre, Victoria Street). No block signals.
- **4-aspect colour-light aspects** (`RED` / `YELLOW` / `DOUBLE_YELLOW` / `GREEN`) derived by a pure adjacent-starter cascade; each starter held RED until its station is served. "Section occupied" is proxied by **a station with an unserved booked stop** (single player; no second train).
- **AWS**: a non-green magnet on approach → caution + sunflower → driver acknowledges within a `dt`-counted window, else a full-service penalty brake. CLEAR/WARNING/(braked) machine grown behind `tickAws`, returning `PenaltyReason "AWS"`. **Serving a station** rings the clear bell → sunflower BLACK (R3-1).
- **TPWS**: TSS (train-stop, passing a red starter unserved) and OSS (overspeed at a loop ahead of a red starter) → `PenaltyReason "TPWS"`.
- **SPAD** detection (front passing a RED starter not served) → asserts `"TPWS"` (the TSS train-stop *is* the same physical crossing) plus a sticky `spad` flag; **no new union member**.
- **Latch enabling change (P2-RA, KEEP)**: scope the `tickSafety` carry-forward seed (`controls.ts:258`) to **DSD-only**, so AWS/TPWS reasons live in the set **iff `tickAws` returns them this tick**. The release machine stays untouched.
- **HUD**: current signal aspect + an AWS sunflower indicator (new `HudView` fields; two new DOM lamps).
- **Pure tested core**: all AWS/TPWS/aspect logic advances off explicit `dt`; nothing under `src/sim` uses `Date`/`Math.random`/`performance.now`/`setTimeout`. Penalty arrives **only** as a `PenaltyReason` through the latch (full service) — never via `SimInputs.emergency`.

### Non-goals (deferred — do not build)
Interlocking · route-setting · junctions/points · second train · ATP/ETCS · timetable/dwell engine (no `dwellUntil`) · real block-occupancy detection (served-stop proxy) · audio from `src/sim`. `SimInputs.dir` is **not** widened to neutral. No env vars, feature flags, or build steps. **No one-tick BRAKE-release lag, no `releaseArmed` field. No block signals. No terminus starter.**

**`served` is a one-way "booked stop done" flag (deferred re-arm).** Monotonic add-only: once served it stays served for the run. Reversing back and re-approaching a served station does **not** re-arm its starter. This is the realistic "booked stop is done" semantics for a single-player one-shot run.

## 2. The design

### 2.0 Shape, derived from the oracles (testability-first)

The oracle list (§4) demands exactly **four pure functions** plus **one threaded state record**:

| Oracle group | Function under test | Inputs it forces |
|---|---|---|
| **A** aspect cascade | `aspectAt(route, i, served)` | route starters + served-set, **no train, no `dt`** |
| **B** AWS warn→ack / warn→penalty / serve-resets / brake-outranks | `tickAws` phase machine | magnet crossing, serve, `intent.acknowledge`, `dir`, `dt` |
| **C** penalty: ack-at-stand drops the reason **same tick** (no lag); ack-while-moving keeps it | `tickAws` latch lifecycle + DSD-scoped seed + fold order | reason cleared the same tick on ack-at-stand; re-asserted while moving |
| **D** TPWS TSS / **F** SPAD | `tickAws` crossing test | red starter crossed unserved |
| **E** TPWS OSS | `tickAws` crossing test | OSS loop crossed at overspeed behind a RED starter |
| **G** dt-slice independence (STEADY only) | all of the above | synthetic prev/now positions, coarse vs fine `dt` end-state equality on a STEADY interval + within-tick precedence |
| **H** O13 migration | `tickAws` stub→real | empty-signal route ⇒ no reasons |

Everything timed lives in **`AwsState`**, owned by `aws.ts`, threaded by `main.ts`. `SafetyState` absorbs **no** AWS internals — only `PenaltyReason` crosses the boundary.

### 2.1 Route signal/AWS data model (`route.ts`)

Signals are **point features**, not piecewise-constant fields — their own sorted array and point queries. **Do not** force them into `Segment<T>` (the M8-on-ramp trap). Aspects are **derived**, never stored.

```ts
// route.ts — NEW
export type Aspect = "RED" | "YELLOW" | "DOUBLE_YELLOW" | "GREEN";

export interface Signal {
  /** Signal-post chainage, m. The single stored chainage.
   *  Starter authoring rule = station stop-board chainage + STARTER_OFFSET
   *  (post sits JUST PAST the platform). */
  chainage: number;
  /** Station whose unserved booked stop holds this starter RED.
   *  Every signal is a station starter; there are no plain block signals. */
  protects: string;
}

export interface Route {
  length: number;
  stations: Station[];
  grades: Segment<number>[];
  speedLimits: Segment<number>[];
  curvatures: Segment<number>[];
  signals: Signal[]; // NEW — required; sorted ASCENDING by chainage (authoring invariant)
}
```

- **One chainage per signal post.** The AWS magnet (`chainage − AWS_MAGNET_OFFSET`) and OSS loop (`chainage − OSS_LOOP_OFFSET`) are **derived** at use-site, never stored. Constants live in `aws.ts`; **`STAND_EPS` is not overloaded**.
- `signals` is **required** (house style, cf. `SimInputs.emergency`). The three synthetic fixtures get `signals: []`, keeping every Phase-0/1 oracle green and making the O13 migration trivial.
- **No `blockSections`** interval data — the served-stop proxy needs none (D7). **No block signals at all** (R3-1).

Pure point queries in `route.ts`, mirroring `lookup`'s guarded style:

```ts
/** Nearest signal AHEAD of s in travel direction dir, or null. */
export function nextSignalAhead(route: Route, s: number, dir: 1 | -1): { i: number; sig: Signal } | null;

/** Indices of any positions in `points` lying in the forward swept interval
 *  (sPrev, sNow]. Generic over an arbitrary position array so one query serves
 *  signal posts, AWS magnets, and OSS loops alike. */
export function pointsCrossedFwd(points: number[], sPrev: number, sNow: number): number[];
```

`pointsCrossedFwd` is purely geometric over a number array, so the caller queries three derived arrays in one place — signal posts, posts−`AWS_MAGNET_OFFSET`, posts−`OSS_LOOP_OFFSET` — and returned indices map straight back to signal index `i` (the arrays are index-aligned and full-length, now over **3 starters**). It only handles the forward case (`sNow > sPrev`); reverse/standing crosses nothing. End-clamp-safe and array-index-guarded for `noUncheckedIndexedAccess`.

**WESTFORD_EASTBANK authoring (V4 — `STARTER_OFFSET = 120`, STARTERS ONLY).** Each *intermediate* served station gets a starter just past its platform. **Westford (origin, never a booked arrival) and Eastbank (terminus) get no starter** (R3-2): the terminus has no onward section, so a post there would clamp to the stop board / buffer and self-SPAD — it is omitted entirely. So there are exactly **three** starters. With `STARTER_OFFSET = 120`, `AWS_MAGNET_OFFSET = 180`, `OSS_LOOP_OFFSET = 50`:

| Signal | protects | stop board | post chainage | magnet (post−180) | OSS loop (post−50) | rationale |
|---|---|---|---|---|---|---|
| S1 | Riverside | 1500 | **1620** (=1500+120) | 1440 | 1570 | magnet in the 25 mph Riverside approach (1400–1600) |
| S2 | City Centre | 3100 | **3220** (=3100+120) | 3040 | 3170 | **long run-in** (S1→S2 ≈ 1600 m) → believable G→YY→Y→R; magnet in 20 mph City curves (3000–3200) |
| S3 | Victoria Street | 4400 | **4520** (=4400+120) | 4340 | 4470 | magnet in the 25 mph Victoria approach (4300–4500) |

Geometry derivation and checks (R3-2, R3-4 / F4):
- **All three posts sit comfortably inside the route**: max post 4520 < length 6000. No post clamps; the terminus self-SPAD problem is dissolved by *omitting* the Eastbank starter entirely.
- **Geometry invariant `board + platformHalf < post − FRONT_OFFSET` holds for all three:**
  - Riverside: 1500 + 90 = **1590 < 1618** = 1620 − 2 ✓
  - City Centre: 3100 + 100 = **3200 < 3218** = 3220 − 2 ✓
  - Victoria Street: 4400 + 90 = **4490 < 4518** = 4520 − 2 ✓

  Therefore a legal stop with the front anywhere in `(board − platformHalf, board + platformHalf]` is **short of the post**. The station serves at the stand → its starter flips GREEN → the front later crosses a GREEN (not RED) post on departure ⇒ **no SPAD/TSS**. This is the keystone.
- **Magnet PSR sanity**: every approach magnet lands in a sensible posted-speed-restriction zone — S1 1440 ∈ [1400,1600] 25 mph; S2 3040 ∈ [3000,3200] 20 mph; S3 4340 ∈ [4300,4500] 25 mph. All sit in the low-speed run-ins where a caution is realistic.
- The S1→S2 ~1600 m gap is the one section long enough that double-yellow is *not* decorative; the S2→S3 pair shows only R/Y/G.

### 2.2 Aspect rule (pure, train-free)

A starter `i` is **held RED** iff its `protects` station is **not in `served`**. Otherwise it cascades off the **next adjacent starter** ahead (a clean adjacent-starter cascade, one R→Y→YY→G rung per starter; with block signals removed there are no transparent links to thread through). `served` lives in `AwsState` (§2.3). Test own held-red FIRST, then look ahead:

```ts
// route.ts — pure, O(1) per signal amortised, no train, no dt
export function aspectAt(route: Route, i: number, served: ReadonlySet<string>): Aspect {
  const sig = route.signals[i];
  if (!sig) return "GREEN";                                   // guard (noUncheckedIndexedAccess)
  if (!served.has(sig.protects)) return "RED";                // OWN held-red first
  const ahead = route.signals[i + 1];                         // next starter toward Eastbank
  if (!ahead) return "GREEN";                                 // last starter, nothing ahead
  const aheadAspect = aspectAt(route, i + 1, served);         // backward sweep, depth ≤ n
  if (aheadAspect === "RED") return "YELLOW";
  if (aheadAspect === "YELLOW") return "DOUBLE_YELLOW";
  return "GREEN";
}
```

Every signal is a starter (`protects !== ""` always), so the own-held-red test reduces to `!served.has(sig.protects)`. Recursion depth bounded by `signals.length` (3); index strictly increases ⇒ no cycles; terminates at the last starter (A4 pins the guarded out-of-range read). Double-yellow is reachable: with only S3 unserved (S1, S2 served), the cascade gives S3=RED, S2=YELLOW, S1=DOUBLE_YELLOW.

**Served-set writer.** `served` grows in the `tickAws` path: a station enters `served` the first tick the front is within `±platformHalf` of its stop board **and** `|speed| ≤ STAND_EPS`. The only writer; position-only proxy, not a dwell model (Residual #2); **monotonic add-only**; the one `dt`-sensitive write (G-note, §4 G).

### 2.3 `AwsState` + the new `tickAws` signature and state machine

`prevChainage` is passed **explicitly** (not folded into `AwsState`); `dir: 1 | -1` is passed **explicitly**: at a stand `sign(state.speed)` is undefined, so heading comes from the same authority `resolveInputs` uses, `ControlState.lastDir`. `main.ts` derives `dir` from `controls.lastDir`; `tickAws` stays pure on its explicit arguments.

```ts
// aws.ts — owned here, threaded by main.ts; pure, dt-counted
export type AwsPhase = "CLEAR" | "WARNING"; // warning lifecycle only; "braked" derived from brakeReason
export type Sunflower = "BLACK" | "CAUTION"; // all-black = clear; black/yellow = caution-latched

export interface AwsState {
  phase: AwsPhase;        // AWS warning lifecycle: CLEAR | WARNING
  warnTimer: number;      // s remaining in the ack window; counts DOWN off dt (WARNING only)
  sunflower: Sunflower;   // latched caution reminder; reset to BLACK on serving a station
  brakeReason: PenaltyReason | null; // single BRAKE-latch source: "AWS" | "TPWS" | null;
                          //   braked ⟺ brakeReason !== null (F6: the one source of "braked")
  served: ReadonlySet<string>;       // stations whose booked stop is done (occupancy proxy)
  spad: boolean;          // sticky SPAD event flag (HUD/scoring)
}

export interface AwsHud { sunflower: Sunflower; spad: boolean; } // NO aspect field here
export interface AwsOutput { reasons: PenaltyReason[]; }         // UNCHANGED latch-boundary type

export function createInitialAws(): AwsState {
  return { phase: "CLEAR", warnTimer: 0, sunflower: "BLACK", brakeReason: null,
           served: new Set(), spad: false };
}

export function tickAws(
  aws: AwsState,
  state: SimState,
  route: Route,
  intent: ControlIntent,
  prevChainage: number,
  dir: 1 | -1,          // explicit heading (from ControlState.lastDir in main.ts)
  dt: number,
): { next: AwsState; reasons: PenaltyReason[]; hud: AwsHud };
```

**F6 — collapsed redundant phase (V4 simplification).** V3 carried a 3-value `phase: "CLEAR" | "WARNING" | "BRAKE"`, but `phase === "BRAKE"` was exactly `brakeReason !== null` — two fields that could drift. **`phase` is now `"CLEAR" | "WARNING"` (the warning lifecycle only); "braked" is derived as `brakeReason !== null`.** A single source of truth: anywhere the V3 machine read `phase === "BRAKE"`, the V4 machine reads `brakeReason !== null`. No field can disagree with itself.

**No `releaseArmed`, no one-tick lag.** The DSD-scoped carry-forward seed (P2-RA) already makes AWS/TPWS releasable. The genuine "survive ack" requirement is **ack-WHILE-MOVING**, not "survive a stand-ack while still demanding." So an ack-at-stand while braked clears the reason **the same tick**; an ack-while-moving never clears it. This removes a whole field and a whole transition — simpler to hold in the head.

**One BRAKE latch path for AWS + TSS + OSS.** A single `brakeReason: "AWS" | "TPWS" | null` carries the latched demand (`braked ⟺ brakeReason !== null`). Both an AWS-deadline miss and a TPWS TSS/OSS trip set `brakeReason`; both re-assert their own reason every braked tick and both clear identically (D5).

#### The state machine (one tick, all `dt`-counted)

Front position uses fixed `FRONT_OFFSET` (legacy 2.0 m) **in travel direction**: `frontNow = state.chainage + dir·FRONT_OFFSET`, `frontPrev = prevChainage + dir·FRONT_OFFSET`. **All edge logic fires only on strict forward motion `frontNow > frontPrev`** — reverse/standing crosses nothing (D10).

1. **Served update**: if `|speed| ≤ STAND_EPS` and front within `±platformHalf` of a station, add it to `served` (copy-on-write, add-only). Clears that station's starter (relaxes the cascade behind it). **On a station newly entering `served`, ring the clear bell: `sunflower = "BLACK"` (R3-1).** Serving is a forward, always-reachable event, so the sunflower reset is reachable on every normal stop without any block signal.
2. **Magnet crossings** (`pointsCrossedFwd` over posts−`AWS_MAGNET_OFFSET`): for each magnet crossed, evaluate that signal's `aspectAt` over the (possibly just-updated) `served`:
   - **GREEN magnet** (a served station's starter, e.g. when re-passing) → `sunflower = "BLACK"`; if `phase === "WARNING"`, cancel to `CLEAR`. Does **not** clear a brake (F6: guarded on `brakeReason === null`).
   - **non-GREEN magnet** → only if `brakeReason === null` (F6): `phase = "WARNING"`, `warnTimer = AWS_WARN_WINDOW`, `sunflower = "CAUTION"`. A latched brake outranks a fresh magnet warning.
3. **WARNING countdown**: if `phase === "WARNING"`:
   - `intent.acknowledge` (allowed **while moving** — D4) ⇒ `phase = "CLEAR"`, sunflower stays `CAUTION` (latched reminder), no penalty.
   - else `warnTimer −= dt`; if `warnTimer ≤ 0` ⇒ `brakeReason = "AWS"` (now braked), `phase = "CLEAR"` (the warning lifecycle is over; "braked" is carried by `brakeReason`).
4. **TPWS / SPAD** (`pointsCrossedFwd` over the signal-post array, evaluating `aspectAt` at the crossing):
   - **TSS / SPAD**: a starter crossed while `aspect === "RED"` (station **unserved** — a blow-through) ⇒ `brakeReason = "TPWS"`, `spad = true`. A served station's starter is GREEN, so a normal stop crosses nothing red — SPAD-free.
   - **OSS**: the signal's OSS loop crossed while that signal `aspect === "RED"` **and** `|speed| > OSS_TRIP_SPEED` ⇒ `brakeReason = "TPWS"`.
   - **Within-tick precedence (F6/G2)**: when a single coarse `dt` sweeps past *both* a magnet (step 2) and its red starter (step 4), AWS is evaluated first (may set `brakeReason="AWS"`), then TPWS/TSS **overwrites `brakeReason` to `"TPWS"`** and sets `spad`. Deterministic order: *served → magnet/AWS → TPWS/SPAD → release*. TSS/SPAD is authoritative (the train physically passed the red), so `"TPWS"` wins; the latch behaviour (full-service brake) is identical either way.
5. **Braked clearing — NO LAG.** While `brakeReason !== null`:
   - If `intent.acknowledge && |speed| ≤ STAND_EPS` (**ack at a stand**): `brakeReason → null` **this same tick**; sunflower stays `CAUTION` until the next serve. Step 6 then returns `[]`, so the DSD-scoped seed carries nothing forward and the latch releases the same tick (C2/C2-PIN).
   - Else (ack-while-moving, or no qualifying ack): **re-assert** `brakeReason`, stay braked. Step 6 returns `[brakeReason]`; the fold re-adds it, so it **survives** (C4). This is the real "survive ack" case.
   - (`spad` stays sticky — a scored event.)
6. **Reasons assembled**: `reasons = brakeReason ? [brakeReason] : []`.
7. **HUD**: return `{ sunflower, spad }` **only**. The HUD aspect is **not** computed here; `buildHudView` derives it from `served` via `nextSignalAhead` + `aspectAt` (§2.6).

All countdowns are `dt`-subtractive and all crossings use swept intervals. Coarse and fine `dt` reach equal end-states for **STEADY intervals** — those in which `served` does not change and no warning *starts* (R3-3); see §2.8 for the precise determinism claim. A coarse `dt` jumping a magnet *and* its starter in one tick is handled because steps 2 and 4 consult the same swept interval, in the F6/G2 order.

**Ack split (D4).** The WARNING-window ack (cancel-before-penalty) is internal to `tickAws` and allowed while moving; the latch-release ack (clearing an *asserted* brake reason) is ack-at-stand. **Both — plus the DSD release in `tickSafety` — consume the single `intent.acknowledge` edge.** One driver acknowledge that cancels the AWS warning, releases a stand-still penalty, and resets DSD is **intended and realistic**. No separate ack path.

### 2.4 How reasons survive / clear with no lag (load-bearing)

This is the keystone mechanism. It depends on **two** facts in `tickSafety`: the carry-forward seed at `controls.ts:258` is **scoped to DSD-only** (P2-RA), and the fold at `:275` runs **before** the DSD-only delete at `:278-281`.

**The seed scoping (the one-line edit at `:258`):**

```ts
const reasons = new Set<PenaltyReason>();
if (s.penaltyReasons.has("DSD")) reasons.add("DSD");
```

This replaces the original `const reasons = new Set<PenaltyReason>(s.penaltyReasons);`. After this edit, AWS/TPWS reasons are **never seed-carried** — re-derived **fresh from the fold** (`for (const r of aws.reasons) reasons.add(r)`, `:275`) every tick. Present in the set **iff `tickAws` returns them this tick**.

- **Survives ack-WHILE-MOVING (C4).** While braked (`brakeReason !== null`) and the driver acks at speed, `tickAws` keeps returning `[brakeReason]` (step 5's else branch re-asserts). The fold (`:275`) re-adds it; the DSD-only delete at `:278-281` cannot touch it. So an ack-while-moving demanding a brake does **not** strand-clear AWS/TPWS — the source still demands it, exactly the pre-authorised clause (`controls.ts:244-246`). This is the genuine "survive ack" case.
- **Clears on ack-AT-STAND the SAME tick (C2).** On an ack-at-stand while braked, `tickAws` sets `brakeReason → null` and returns `[]` **that same tick** (no lag). The fold adds nothing; **because the seed is DSD-scoped** the reason is not seed-carried, the set empties → the latch releases the same tick. **This requires the `:258` seed-scoping**: under the original full carry-forward seed the reason would re-seed every subsequent tick from `s.penaltyReasons` and never be deleted (the release names only "DSD"), making the latch un-releasable forever.
- **DSD coexistence (C-COEXIST).** DSD + an AWS/TPWS reason both latched: an ack-at-stand deletes `"DSD"` (release machine) *and* `tickAws` stops returning AWS/TPWS the same tick (fold adds nothing) ⇒ **both clear together** under the single acknowledge — acceptable, one driver acknowledgment.
- **Why the release machine stays untouched.** The `:280` comment ("Phase-2 persisting reasons would re-add themselves above") assumed fold-sourced reasons are **not** seed-carried. The seed-scoping makes that true: every non-DSD reason in the set is, by construction, this-tick fold output, and drops automatically when the fold stops producing it. So the release machine (`:277-281`), `resolveInputs`, `reduceControls` need **no edit** — only the seed is scoped.
- **`tickAws` only ever touches its own reasons** — never `"DSD"` (R1), and DSD logic never touches `"AWS"`/`"TPWS"`.

> **`tickSafety` release machine + signature unchanged; the carry-forward seed scoped to DSD-only (one line, `:258`).** Survive (ack-while-moving) = the fold re-adds the reason every tick `tickAws` still demands it; clear (ack-at-stand) = `tickAws` stops demanding it the same tick **and** the DSD-scoped seed no longer carries it ⇒ it drops. No `releaseArmed`, no lag.

### 2.5 `tickSafety` + frame order + test-helper ripple

- **`tickSafety` release machine + signature are UNCHANGED; the carry-forward seed is scoped to DSD-only (one line, `:258`).** It still takes `aws: AwsOutput` and folds `aws.reasons` at `:275`. The release machine (`:277-281`), `resolveInputs`, `reduceControls` are **byte-for-byte untouched** — they inherit notch-0 + full-service + power-to-OFF for AWS/TPWS via `penaltyActive`. The **only** change inside `tickSafety` is narrowing the seed at `:258`.

- **Frame reorder (IMPORTANT).** The **current** `main.ts` runs `tickSafety(... tickAws(...))` **before** `step()`. Phase 2 needs **prev→now chainage** for crossing detection, so the frame is **reordered** so `tickAws`/`tickSafety` run **AFTER** `step()`: capture `prevChainage = state.chainage` *before* `step()`, run `step()` to advance, then pass the **pre-step** `prevChainage` and the **post-step** `state.chainage` into `tickAws`. The reorder is explicit:

  ```ts
  let aws = createInitialAws();
  // per frame:
  const intent = intentFromKeys(edges);
  controls = reduceControls(controls, intent, state, safety); // safety = prior frame's
  const inputs = resolveInputs(controls, safety, ADHESION.wetNight);
  const prevChainage = state.chainage;          // pre-step
  state = step(spec, route, state, inputs, dt); // advance FIRST (was after tickSafety in current code)
  const dir = controls.lastDir;                 // same authority resolveInputs uses
  const awsOut = tickAws(aws, state, route, intent, prevChainage, dir, dt); // post-step chainage
  aws = awsOut.next;
  safety = tickSafety(safety, intent, state, dt, { reasons: awsOut.reasons });
  scene.render(state.chainage);
  hud.update(buildHudView(state, controls, safety, route, aws.served, awsOut.hud));
  ```

  **One-frame implication:** because `tickSafety` now runs after `step()`, an AWS/TPWS penalty asserted this frame inhibits power/forces brake from **the next** `resolveInputs` (next frame), not the same frame. This is identical to the existing one-frame staleness `reduceControls` already reads (`safety = prior frame's`), so the latch is no less responsive than DSD today. Determinism is preserved within steady intervals (§2.8); `tickAws` is pure on explicit `prevChainage`/`dir`/`dt`; the reorder only changes *which* chainage pair it sees, not the integrator.

  `main.ts` stays arithmetic-free (the untested shell). The richer `tickAws` return is destructured to `{ reasons }` at the latch boundary, so only `PenaltyReason` crosses into the latch module (D2).

- **`test/controls.test.ts` ripple** (exact):

### 2.8 Determinism claim, narrowed (R3-3)

**Exact dt-slice independence does NOT hold for every interval.** It fails precisely when, within the swept interval, **`served` changes** or **a WARNING starts**:
- `served` is updated (step 1) *before* the swept magnet/starter crossings (steps 2/4). A coarse tick that serves a station and *then* re-reads an earlier magnet diverges from fine slices that read the magnet before serving — the magnet's evaluated aspect (RED vs GREEN) depends on whether the serve had already landed.
- `warnTimer` depends on **where in the swept interval** the magnet lay: a coarse tick that crosses the magnet partway through subtracts the whole `dt` from `warnTimer` as if the magnet were crossed at the interval start, shifting the warn countdown's phase by up to one slice versus fine slicing.

**Why this is safe in practice (no event-time interpolation needed).** The runtime clamps `dt` to `MAX_DT = 0.05` (`simulation.ts`), so a real frame advances at most ~`v·0.05` m. A magnet sits `STARTER_OFFSET − AWS_MAGNET_OFFSET`... — more concretely, the magnet (post−180) and the post are 180 m apart, and the serve point (within `±platformHalf` of the board) and the post are ≥ `STARTER_OFFSET − platformHalf` ≥ 20 m apart, all far larger than one clamped frame's travel at any plausible line speed. **A real clamped frame never spans both a magnet and a serve, nor both a magnet and its post.** So the divergent intervals never arise at runtime; adding event-time interpolation would be over-engineering for this scope and is explicitly **not** done.

**The precise claim (what G1/G2 test).** dt-slice independence holds for **STEADY intervals**: those in which `served` does **not** change and no WARNING **starts** within the interval. G1/G2 test independence on a STEADY fixture (a held-RED, **never-served** starter; the train either already past the warn transition or never reaching it), so the two `dt`-sensitive writes (served-flip, warn-start) are excluded by construction. Served-changing and warning-starting intervals are **explicitly excluded** from the claim, justified by the `dt` clamp above (G-NOTE).

## 5. Residual risks

1. **Fold-before-delete order + seed-scoping coupling (highest).** Persistence/release rests on two coupled facts: the seed at `:258` is **DSD-scoped**, and the fold (`:275`) runs **before** the DSD-only delete (`:278-281`). The un-releasable-latch risk is resolved by the seed-scoping; the survive-ack risk is resolved by `tickAws` re-asserting while moving (no lag needed). What remains fragile is **fold-order**: moving the fold after the delete, or generalising the delete to non-DSD reasons, would mis-clear mid-demand. C2/C2-PIN/C4/C-COEXIST are the guards; comments cite `:258`, `:275`, `:278-281`.
2. **Served-set as occupancy proxy is a fiction, and is `dt`-sensitive.** Aspect colour couples to "drove well enough to stop." The exact served-flip tick depends on `dt` slicing (G-NOTE) — pinned, defined, excluded from the dt-independence claim. A train crawling through a platform without truly stopping could mis-flag. If gamey, swap `served` for a scripted held-signal list with zero cascade changes.
3. **Double-yellow only on the S1→S2 section.** A1 proves the logic; the route's *use* of YY is decorative-but-correct (the S2→S3 pair shows only R/Y/G).
4. **OSS abstraction.** Speed-threshold-at-a-loop-chainage, not real loop-spacing/arm-timer physics. Trip speed is one tunable constant.
5. **OSS loop placement.** Each OSS loop is `post − OSS_LOOP_OFFSET(=50)`, sitting just behind its starter, so it is reached (and can arm) on the approach **before** the starter it protects. All three OSS loops (1570, 3170, 4470) lie ahead of their derived magnets and behind their posts — correct arm-then-trip ordering.
6. **Direction reconciliation.** Heading comes from explicit `dir = ControlState.lastDir`, not `sign(speed)`. All crossings gate on strict `frontNow > frontPrev`; D2/G2 guard reverse and coarse-dt.
7. **`prevChainage` + `dir` + frame-reorder plumbing is in the impure shell.** A wiring bug (post-step chainage as prev, wrong `dir`, or not capturing prev before `step()`) is untestable at `main.ts`. Mitigation: `tickAws` is pure on its explicit `prevChainage`/`dir`, so the *logic* is fully tested (G1 constructs the pair directly); only the plumbing is trust-based.
8. **Starter geometry (resolved).** `board + platformHalf < post − FRONT_OFFSET` for all three starters, so a normal stop anywhere in-platform serves before the front reaches the RED post. D1-SERVED and D-OVERSHOOT pin it across all three.
9. **Reverse-and-re-approach (out of scope).** `served` is monotonic add-only.
10. **No terminus protection (accepted).** Eastbank (terminus) has no starter — there is no onward section, so a post there would self-SPAD. The buffer-stop run-in (15 mph PSR) and the route-length clamp in `step()` handle the end-of-line; no signalling guards it. This is realistic for a buffer-stop terminus.
