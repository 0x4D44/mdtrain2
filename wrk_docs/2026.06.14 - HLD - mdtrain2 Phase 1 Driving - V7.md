## 1. Scope & non-goals

### In scope (control + HUD layer only)
- A **pure controls/safety reducer** (`src/sim/controls.ts`) holding `ControlState` + `SafetyState`, exposing the testable functions `reduceControls`, `tickSafety`, `resolveInputs`, and `buildHudView` that turn driver intent + timed safety into the existing `SimInputs` for `step()`.
- **Prototypical stepped controls:** notched POWER (OFF + 4 notches), stepped BRAKE (RELEASE + 3 service applications — STEP 1, STEP 2, FULL SERVICE — + EMERGENCY past an end gate; index 3 **is** FULL SERVICE), REVERSER (FORWARD/OFF/REVERSE).
- **Interlocks:** no power against an applied brake; reverser changes only with power off and at a stand; reverser OFF inhibits power.
- **Safety now:** the DSD/vigilance device (timer flavour) feeds **one shared penalty latch + one acknowledge**; the DRA is a **separate latching power-inhibit** (no brake, not a penalty reason).
- **One surgical pure-physics change:** a real, distinct EMERGENCY deceleration (`brakeEmergencyDecel=1.3`) selected in `brakeForce`, with its own analytic oracle — **not** faked as `brake=1`.
- A **keymap→intent input layer** kept **inline in `main.ts`** (decision-free ~20-line lookup; no separate module).
- A **structured DOM HUD** split into a pure `buildHudView` (tested) + a thin `createHud` DOM writer (untested).
- A **typed AWS/TPWS no-op stub** at `src/sim/aws.ts` defining the Phase-2 seam and passing through unchanged.
- A thin, deliberately-untested `main.ts` wiring keyboard → controls → safety → `resolveInputs` → `step` → `render` → HUD.

### Non-goals (explicit — do not build)
- 3D cab interior, lighting, GTO audio (Phase 3); `scene.ts`, `index.html` 3D world unchanged.
- AWS/TPWS *behaviour* — magnets, deadlines, sunflower, SPAD (Phase 2). Phase 1 ships only the seam + no-op stub.
- Gamepad/touch input (Phase 5); keyboard only.
- Route signal/AWS-magnet positions, station-stop scoring (no data; later phases).
- Pneumatic pipe-propagation, per-axle brake modelling, continuous (analog) levers.
- Widening `SimInputs.dir` to include neutral `0` (breaking — `dir` stays `1|-1`; neutral lives only in the control layer).
- New state on `SimState`; latches/timers inside `step()`/`physics.ts`.
- Exposing `mu` to the driver — stays hardwired `ADHESION.wetNight`. No env vars, no feature flags, no CI changes, no new runtime deps.

## 2. The design

### 2.0 Risk-first overview — the five failure modes, designed away

This table leads because the rest of the design exists to defeat these. Each row maps a failure mode to the single design move that prevents it; the oracle that pins it is in §4.

| # | Failure mode | How it bites | Design move that prevents it | Oracle |
|---|---|---|---|---|
| **R1** | **Shared latch fights itself** (DSD now, AWS/TPWS later) | Two systems each "own" a latch; one clears it while the other still demands brake; or ack clears DSD but AWS re-latches next tick → un-releasable or prematurely-released brake | **One latch = `Set<PenaltyReason>`, active iff non-empty.** Each subsystem only adds/removes *its own* reason. Documented per-reason clearability predicate (§2.6). Ack is edge- and stand-gated. | O9–O12 |
| **R2** | **Interlock composition order** | "No power vs brake", DRA, penalty, reverser-OFF all mutate `notch`/`brake`; wrong order → power leaks through a penalty, or a released DRA still blocks | **One pure `resolveInputs`** with a fixed pipeline: power-inhibitors compose by OR, brake-demands by `max`. Order-independent by construction; each inhibitor pinned in isolation. | O3–O5, O12, O15a–e |
| **R3** | **Emergency change breaks existing oracles** | The braking-distance oracle (`physics.test.ts:109`) drives `brake:1` and asserts `brakeServiceDecel`. Reinterpreting `brake=1` as emergency breaks it. | Keep `brake∈[0,1]` scaling **service** decel. Add a **separate `emergency` channel** through `AccelInputs`/`SimInputs`→`brakeForce`, default `false`. | O1–O2, regression |
| **R4** | **Wall-clock / determinism leak** | A `performance.now()`/`setTimeout` in DSD → non-reproducible, frame-rate-dependent vigilance | **All timed safety advances off the explicit `dt`** passed to `tickSafety`. No `Date`/`performance.now`/`setTimeout`/`Math.random` anywhere under `src/sim` (the keymap is inline in `main.ts`, whose rAF `dt` is the one allowed wall-clock — D10; there is no `src/input` directory). Verifiable by grep + a frame-rate-independence oracle. | O8 |
| **R5** | **`brakeActual=1` start mismatch** | `createInitialState` seeds `brakeActual=1`. If the brake controller starts at RELEASE, HUD shows a released lever against a fully-applied brake and frame 1 could roll. | **`createInitialControls()` seeds `brakeStep = FULL SERVICE`** so lever, demand, and `brakeActual` all agree at spawn. | O3 (no-creep) |

**On the stale speed read (R2/R5):** the interlocks (reverser-at-stand, DRA-at-stand, motion-gated vigilance, latch release) read the **pre-step**, one-frame-stale `state.speed` — the value from before this frame's `step()` runs. **What the through-zero snap guarantees, exactly:** when a notch is suppressed (an inhibitor fires) *and* speed is at or crossing zero, `step()`'s through-zero snap zeroes the speed, so a one-frame-stale "at a stand" reading can never let **power-induced** motion leak through — that is the no-creep guarantee, and nothing more. It does **not** claim that a brake-released, power-inhibited train will stay put on a gradient: such a train can roll under gravity, which is correct physics governed by the **brake**, not by the power interlock. The snap suppresses power-induced creep at/through zero; it is not a parking brake.

**Why one pure brain (`controls.ts`) + thin shell.** The whole control/safety/HUD-derivation logic is a single pure module under `src/sim/`, unit-tested with the same discipline as `physics.ts`. ### 2.2 Core types

```ts
// src/sim/controls.ts
export type Reverser = "FWD" | "OFF" | "REV";
export type PenaltyReason = "DSD" | "AWS" | "TPWS"; // AWS/TPWS reserved for Phase 2

export const POWER_NOTCHES = 4;                          // OFF + 4 detents
export const BRAKE_SERVICE_STEPS = 3;                    // service steps 1..3
export const BRAKE_FULL = BRAKE_SERVICE_STEPS;           // index 3 = FULL SERVICE
export const BRAKE_EMERGENCY = BRAKE_SERVICE_STEPS + 1;  // index 4 = EMERGENCY (past the end gate)
export const STAND_EPS = 0.3;   // m/s — "at a stand" (reverser, DRA, latch release)
export const DSD_PERIOD = 60;   // s vigilance interval
export const DSD_WARN_WINDOW = 7; // s of warning before penalty

export interface ControlState {
  powerNotch: number;   // 0..POWER_NOTCHES   (integer index; 0 = OFF)
  brakeStep: number;    // 0..BRAKE_EMERGENCY (0=RELEASE, 1=STEP 1, 2=STEP 2, 3=FULL SERVICE, 4=EMERGENCY)
  reverser: Reverser;
  lastDir: 1 | -1;      // remembered heading; reverser OFF keeps this, feeds SimInputs.dir; never 0
  dra: boolean;         // Driver's Reminder Appliance: set ⇒ power inhibited
}

export interface SafetyState {
  vigilanceTimer: number;           // s remaining before warning; counts DOWN off dt
  dsdWarning: boolean;              // warning window active (amber lamp)
  penaltyReasons: ReadonlySet<PenaltyReason>; // the SHARED latch: active iff non-empty
}

export function createInitialControls(): ControlState {
  return { powerNotch: 0, brakeStep: BRAKE_FULL, reverser: "OFF", lastDir: 1, dra: false };
  //                                  ^ FULL SERVICE at spawn ⇒ matches brakeActual=1 (R5)
}
export function createInitialSafety(): SafetyState {
  return { vigilanceTimer: DSD_PERIOD, dsdWarning: false, penaltyReasons: new Set() };
}
```

**Integer indices are the source of truth** for power/brake (not floats): notch fraction `powerNotch / POWER_NOTCHES` → exactly {0, .25, .5, .75, 1}; service brake fraction is defined explicitly as `brakeFraction(brakeStep) = clamp01(brakeStep / BRAKE_SERVICE_STEPS)` → {0, ⅓, ⅔, 1} for steps 0..3, so **FULL (index 3) → 1** and **EMERGENCY (index 4) → clamp01(4/3) = 1**. That is: **both FULL and EMERGENCY map to brake demand `1`.** The EMERGENCY *physics* difference does not come from the brake fraction at all — it comes solely from the separate `emergency` boolean (§2.5 step 5 / §2.7), which selects `brakeEmergencyDecel`. This clamped definition is load-bearing for oracles O5 and O12. Exact fractions make `notch === 0.5` assertions trivial and avoid float drift. Pure helpers `notchFraction`, `brakeFraction`, `isEmergency`, `brakeApplied`, `penaltyActive(s) = s.penaltyReasons.size > 0` are exported for HUD + tests.

### 2.3 The intent + the inline keymap (in `main.ts`)

`ControlIntent` is a flat record of edges for one frame, applied by the pure `reduceControls`/`tickSafety`:

```ts
// declared in controls.ts (consumed by the pure reducers); produced inline in main.ts
export interface ControlIntent {
  powerUp: boolean; powerDown: boolean;                 // edge: one detent per press
  brakeUp: boolean; brakeDown: boolean;                 // edge
  emergency: boolean;                                   // edge: slam to EMERGENCY
  reverserFwd: boolean; reverserOff: boolean; reverserRev: boolean; // edge
  toggleDra: boolean;                                   // edge
  acknowledge: boolean;                                 // edge: DSD/penalty ack
  vigilancePing: boolean;                               // any control activity this frame ⇒ resets DSD
}
```

`main.ts` maintains a per-frame **edge set** of just-pressed key codes (`keydown` with `if (e.repeat) return` so a hold never auto-repeats a detent; `keyup`/`blur` housekeeping), then builds the intent from a ~20-line literal lookup table (e.g. `KeyW`→powerUp, `KeyS`→powerDown, `KeyD`→brakeUp, `KeyA`→brakeDown, `Backquote`→emergency, `KeyF`/`KeyN`/`KeyR`→reverser FWD/OFF/REV, `KeyL`→toggleDra, `KeyQ`→acknowledge), with `vigilancePing = edgeSet.size > 0`. The table is decision-free data; the only untested logic is the edge-set construction, which is the same handful of lines whether or not the table lives in its own file — so it stays inline (R: simplicity, "don't predict the future").

**Simultaneous opposing/conflicting edges in one frame** are resolved deterministically before reduction (and pinned by O20): `emergency` overrides all other brake edges (it wins outright); an opposing `brakeUp`+`brakeDown` in the same frame cancel (no brake change); an opposing `powerUp`+`powerDown` cancel (no power change); and if more than one reverser edge (`reverserFwd`/`reverserOff`/`reverserRev`) is present in the same frame, all of them are ignored (no reverser change). This makes a frame carrying contradictory edges well-defined rather than order-dependent.

**Vigilance is reset by ANY control edge** (`vigilancePing = edgeSet.size > 0`), the acknowledge key included. Because the acknowledge key is itself a control edge, it sets `vigilancePing` and so resets the vigilance timer (via `tickSafety` step 1, §2.6) **even on a tick where it does not release the latch** (e.g. moving, so no release) — this coupling is intended: any driver input is vigilance activity. Because `keydown` uses `if (e.repeat) return`, simply *holding* a steady power notch produces no further edges — so a driver who makes no fresh input for `DSD_PERIOD` while moving will (correctly, by design) trip the vigilance penalty. This is the intended deadman behaviour, not a bug: vigilance demands periodic positive driver action, not a held lever.

### 2.4 Control reduction (pure, `controls.ts`)

```ts
export function reduceControls(
  c: ControlState, intent: ControlIntent, state: SimState, safety: SafetyState,
): ControlState;
```

`reduceControls` takes the (prior-frame) `safety` so the power handle can be forced to OFF while a penalty is active (the no-lurch interlock below). Per §2.10's per-tick order it reads the one-frame-stale `safety` from before this frame's `tickSafety` runs — the same stale-state read pattern as the interlocks in R2/§2.0; the one-frame lag is harmless (a penalty that latches this frame forces power off next frame, by which point `resolveInputs` has already zeroed the notch via `penaltyOn`, §2.5 step 3).

Applies, in order, returning a new state (mutates nothing). The frame's edges are first conflict-resolved per §2.3 (emergency overrides brake edges; opposing power/brake edges cancel; multiple reverser edges are ignored):

1. **Reverser** edges: a change to FWD/OFF/REV is accepted **only if `powerNotch === 0` AND `|state.speed| ≤ STAND_EPS`** (power must be off *and* the train at a stand); otherwise ignored. FWD sets `lastDir=1`; REV sets `lastDir=-1`; OFF keeps `lastDir`.
2. **Power** edges: `powerNotch = clamp(powerNotch ± 1, 0, POWER_NOTCHES)`. **Then, if a penalty is active (`penaltyActive(safety)`), force `powerNotch = 0`** — the power handle is driven back to OFF and *stays* OFF until the driver makes a fresh `powerUp` edge **after** the penalty has been released. (A `powerUp` edge applied while the penalty is still active is overridden back to 0 by this same step, so a still-latched penalty can never let the notch climb; only a `powerUp` on a frame where `penaltyActive(safety)` is false advances it.) This is the prototypical interlock against a lurch on release: clearing the penalty does **not** restore the pre-penalty notch, so power resumes only by a deliberate fresh demand from OFF.
3. **Brake** edges: `brakeUp` steps `0→1→2→3(FULL)→4(EMERGENCY)`; `brakeDown` decrements one step clamped to `[0, BRAKE_EMERGENCY]`. **EMERGENCY is reachable only by stepping past FULL (an end gate)** — a single `brakeUp` from FULL reaches EMERGENCY; it is *not* reachable by single-keypress overshoot from a partial step. The `emergency` edge slams `brakeStep = BRAKE_EMERGENCY` directly. **EMERGENCY latches:** a `brakeDown` from EMERGENCY(4)→FULL(3) is **rejected unless the train is at a stand** (`|state.speed| ≤ STAND_EPS`); a `brakeDown` from any *service* step (3→2→1→0) decrements freely. So once EMERGENCY is applied it cannot be stepped down while moving — it releases to FULL only at a stand (D18, O19).
4. **DRA** toggle: accepted **only at a stand** (`|state.speed| ≤ STAND_EPS`).

`reduceControls` owns the reverser-at-stand and DRA-at-stand guards, the EMERGENCY-release-at-stand latch, the **power-to-OFF-under-penalty interlock** (D23), and the stepped clamps. It does **not** apply the no-power-against-brake interlock — that is a demand-time gate (the lever can sit in notch 2 while effort is suppressed) and belongs in `resolveInputs`.

**Residual-risk note (reverser at a crawl):** `STAND_EPS` (0.3 m/s) is the "at a stand" threshold; a train can sit below `STAND_EPS` while still genuinely rolling — e.g. coasting down a gradient, where the through-zero snap (which only fires *at/through* zero, and only with `notch === 0`) does **not** apply. So a reverser change is *technically* permitted at a crawl (≤ 0.3 m/s ≈ 0.67 mph) rather than only at a dead stop. This is benign rather than guaranteed-snapped: the reverser-change interlock already requires `powerNotch === 0` (D16), and a reverser change at a crawl is only reachable with a brake applied (otherwise the train would be accelerating, not crawling) — so the worst case is flipping `lastDir` on a near-stationary, braked train, which changes only the *intended* heading of a train that is about to stop anyway and produces no lurch. No extra constant is introduced.

### 2.5 `resolveInputs` — the interlock pipeline (the PRIMARY SEAM)

The single pure function turning control + safety state into `SimInputs`. Fixed, documented composition order; power-inhibitors OR-compose, brake-demands `max`-compose:

```ts
export function resolveInputs(c: ControlState, s: SafetyState, mu: number): SimInputs;
```

```
1. dir          = c.lastDir                                   // always 1|-1; OFF is expressed as zero notch
2. penaltyOn    = penaltyActive(s)
3. powerInhibit = brakeApplied(c.brakeStep)                   // no power vs applied brake
               || c.reverser === "OFF"                        // reverser OFF inhibits power
               || c.dra                                       // DRA inhibits power
               || penaltyOn                                   // penalty inhibits power
4. notch        = powerInhibit ? 0 : notchFraction(c.powerNotch)
5. emergency    = isEmergency(c.brakeStep)                    // ONLY the driver's EMERGENCY lever; penalty ≠ emergency
6. driverBrake  = brakeFraction(c.brakeStep)                  // clamp01(step/3): FULL→1, EMERGENCY→clamp01(4/3)=1
   penaltyBrake = penaltyOn ? 1 : 0                           // penalty = FULL SERVICE demand
   brake        = Math.max(driverBrake, penaltyBrake)
7. return { notch, brake, dir, emergency, mu }
```

Guarantees pinned by oracles: any applied brake OR penalty OR DRA OR reverser-OFF ⇒ `notch === 0`; the driver can never out-rank a penalty (penalty forces full-service brake + zero power regardless of lever positions); **penalty brake is full *service*, not emergency** (`emergency` is reserved for the driver's deliberate EMERGENCY lever — matches the brief's "full-service, latched" penalty).

The safety layer expresses its override purely as `brake`/`notch`/`emergency` floats — exactly the `step()` contract. `step` keeps owning `brakeActual` (the lag); the safety layer never touches it. The *only* new `SimInputs` field is `emergency`, a genuine physics input, not a safety flag.

### 2.6 DSD/vigilance, DRA, and the shared penalty latch (pure, `controls.ts`)

```ts
export function tickSafety(
  s: SafetyState, intent: ControlIntent, state: SimState, dt: number, aws: AwsOutput,
): SafetyState;
```

Order inside `tickSafety` (returns a new state; advances only off `dt` — R4):

1. **Vigilance reset:** if `intent.vigilancePing` → `vigilanceTimer = DSD_PERIOD`, `dsdWarning = false`. Because the acknowledge key is itself a control edge it raises `vigilancePing`, so an ack always resets the vigilance timer here even on a tick where it does not release the latch (intended — any driver input is vigilance activity, §2.3).
2. **Countdown (motion-gated):** else if the train is moving (`|state.speed| > STAND_EPS`), `vigilanceTimer -= dt` and `dsdWarning = vigilanceTimer ≤ DSD_WARN_WINDOW`; when `vigilanceTimer ≤ 0`, **add `"DSD"`** to `penaltyReasons`. **At a stand (`|state.speed| ≤ STAND_EPS`) the timer holds (does not decrement)**, so a correctly parked train is never penalty-braked.
3. **AWS/TPWS merge:** fold in `aws.reasons` (from the stub, §2.8) at the *same* point DSD adds its reason — Phase 1 contributes none.
4. **Latch release (the one ack path):** apply the per-reason clearability predicate below.

**The CLEAR → WARNING → PENALTY state machine and the release edge:**

```
        ping (timer > WARN)                  ack edge AND |speed| ≤ STAND_EPS
        ┌────────────────────────┐                      (releases reason)
        ▼                        │                           ▲
    [CLEAR] ──timer≤WARN──▶ [WARNING] ──timer≤0──▶ [PENALTY: reasons ∋ DSD]
        ▲                        │                           │
        └──── ping (reset) ──────┘                           │
                                                             │
   release a reason  ⟺  ack edge  AND  |speed| ≤ STAND_EPS    │
                                                              ┘
   latch clears  ⟺  penaltyReasons becomes empty
```

**Documented contract (lives as a comment in `controls.ts`, so the Phase-2 author wiring AWS/TPWS cannot break releasability):**

> A reason is removed from `penaltyReasons` on an `acknowledge` edge **AND** `|state.speed| ≤ STAND_EPS`. The overall latch is "released" only when the set becomes empty — that empty-set rule is the mandated invariant. In Phase 1 the only reason is `"DSD"`: its demand is satisfied once the driver acks at a stand (and, the train being stopped, the motion-gated timer holds rather than re-counting), so the ack-at-stand edge clears it.
>
> *Forward-looking note (Phase 2):* a reason whose source still demands brake **after** ack (e.g. an AWS magnet just passed) should additionally fail a "source no longer demands it" predicate and stay in the set; such reasons slot in **without touching the release machine** — one ack at a stand clears all currently-clearable reasons while a still-asserting subsystem keeps its reason and the brake stays applied. Phase 1 does not implement this clause (no persisting reason exists yet).

**Intended deadman behaviour (not a bug).** The vigilance timer is reset by *any* control edge (`vigilancePing = edgeSet.size > 0`, §2.3), the acknowledge key included. Because `keydown` uses `if (e.repeat) return`, holding a steady power notch produces no further edges, so a driver who makes no fresh input for `DSD_PERIOD` while moving will correctly trip the vigilance penalty. This is by design — vigilance demands periodic positive action, not a held lever — and is pinned from both sides: O9 (no input while moving ⇒ penalty) and O18 (periodic pings while moving ⇒ never penalised).

**DRA** is **control state**, not a penalty reason: it only inhibits power (§2.5), applies no brake, and is cancelled manually at a stand. It must not feed the latch (a reminder appliance is not a protection event), so it lives in `ControlState` as a boolean + lamp.

### 2.7 The physics emergency-decel change (surgical, R3)

`brakeForce` is called from exactly one site (`acceleration`) and has no direct unit test today, so the blast radius is one parameter. Add an `emergency` flag selecting the decel target:

```ts
// physics.ts
export function brakeForce(
  spec: TrainSpec, brakeActual: number, mu: number, emergency = false,
): number {
  const decel = emergency ? spec.brakeEmergencyDecel : spec.brakeServiceDecel;
  const demanded = clamp01(brakeActual) * decel * effectiveMass(spec);
  const adhesionLimit = mu * spec.mass * G; // braked on all axles
  return Math.max(0, Math.min(demanded, adhesionLimit));
}
```

`AccelInputs` gains a required `emergency: boolean`; `acceleration()` passes `inp.emergency` to `brakeForce` (both the moving branch and the static-hold branch read the same flag). The static-hold branch (physics.ts ~97–100, the `// Near standstill` block) **does** consult `brakeForce` — it uses the returned `fBrake` to decide whether the train is held (`|fDrive| ≤ fBrake ⇒ held`) — so threading `emergency` in is **not** a no-op there. The correct property is monotonic and safe: because `brakeEmergencyDecel (1.3) > brakeServiceDecel (0.9)`, emergency yields a holding force **≥** service, so a train held under service is **always** held under emergency (never worse); and on a grade steep enough that a service hold would slip, emergency correctly holds it. `simulation.ts`: `SimInputs` gains a required `emergency: boolean`; `step()` threads it into the `AccelInputs` it builds. The `brakeActual` lag logic is unchanged — `emergency` selects the *decel scale*; the build-up still rides `brakeActual` toward the `brake=1` demand per `BRAKE_LAG`.

Default `emergency = false` on `brakeForce` keeps the function call API backward-safe, but per the brief's preference for **required fields + defaults over optionals** under `exactOptionalPropertyTypes`, `AccelInputs.emergency` and `SimInputs.emergency` are *required*. The single mechanical ripple is adding `emergency: false` to the literals in `physics.test.ts` (the `AccelInputs` in the RK4 oracle and every `SimInputs` in `drive(...)` call sites) — no assertion weakened. Why a flag and not `brake=1`: `brake∈[0,1] × serviceDecel=0.9` can never reach `1.3 m/s²`; emergency must be a genuinely distinct deceleration with its own closed-form stop distance `v²/(2·1.3)`.

### 2.8 AWS/TPWS future seam (typed no-op stub)

The cheapest shape Phase 2 can extend without rework — a reasons-shaped output and one pure fold point, homed at `src/sim/aws.ts` (a sibling of `controls.ts`; no `safety/` subdirectory for a single file). No `AwsState`/`armed` machinery beyond what the seam needs:

// src/sim/aws.ts

### 2.9 The structured HUD

`buildHudView` is **pure and lives in `controls.ts`** (so it is unit-tested without any DOM dependency); `createHud` in `src/ui/hud.ts` is a thin, untested DOM writer:

```ts
// buildHudView in controls.ts
export interface HudView {
  speedMph: number; limitMph: number;
  reverser: "FWD" | "OFF" | "REV";
  powerNotch: number; powerMax: number;            // e.g. 2 of 4
  brakeLabel: string;                              // "RELEASE" | "STEP 1" | "STEP 2" | "FULL SERVICE" | "EMERGENCY"
  brakeDemandPct: number; brakeActualPct: number;  // show BOTH — the ~1.5–2 s lag must be visible
  dra: boolean; dsdWarning: boolean; penalty: boolean;
  nextStop: string; chainage: number;
}
export function buildHudView(state, c, s, route): HudView; // PURE, tested (powerMax from POWER_NOTCHES)

// createHud in src/ui/hud.ts
export function createHud(parent: HTMLElement): { update(v: HudView): void }; // DOM, untested
```

**The load-bearing split, stated crisply:** `buildHudView` is pure and **is** tested — mph (`|speed|·2.236936`), next-stop lookup (first station ahead of `chainage` in `dir`; when there is no station ahead — end of line — `nextStop = "— (end of line)"`, and the array lookup is guarded for `noUncheckedIndexedAccess` so the possibly-`undefined` element is handled, never dereferenced blindly), brake label from `brakeStep`, demand-vs-actual %, and the lamp booleans.

**The next-stop boundary rule, stated explicitly:** `buildHudView` picks the next station **strictly ahead** in `lastDir`. A station at the train's **exact current chainage counts as already reached**, not as the next stop — i.e. a station is a candidate only when `(station.chainage − chainage)·lastDir > 0`; any station with `(station.chainage − chainage)·lastDir ≤ 0` is skipped (it is at or behind the train). This is load-bearing at spawn: at chainage 0 with `lastDir = +1`, **Westford** (chainage 0) is already reached, so `nextStop` is **Riverside** (chainage 1 500), not Westford. (Pinned by O14, SPAWN case.)

**Demand vs actual, and the next-stop heading, made explicit.** `brakeDemandPct` shows the **resolved brake demand** — the `brake` value out of `resolveInputs`, i.e. including any penalty- or emergency-forced full application — *not* the bare lever fraction; `brakeActualPct` shows `brakeActual` (the lagged pneumatic build-up). So under penalty the displayed demand reflects the forced full-service even if the lever sits lower, and at spawn both read **FULL SERVICE** (lever = FULL SERVICE per R5, and `brakeActual = 1`). The next-station scan picks "next station ahead" using `lastDir` (the retained heading), so it is well-defined even when the reverser is OFF (which leaves `lastDir` intact, §2.2/§2.4).

**`noUncheckedIndexedAccess` audit — every new array index Phase 1 introduces is guarded.** The next-station scan in `buildHudView` is the **only** new *array* index site: it iterates `route.stations` and the result is `T | undefined`, handled by the end-of-line fallback above (never blind-dereferenced). The other "lookups" are deliberately **not** array indexing and so incur no `noUncheckedIndexedAccess` obligation: the brake label is computed from `brakeStep` by a total `switch`/conditional (RELEASE / STEP 1 / STEP 2 / FULL SERVICE / EMERGENCY), not an array index; the power notch is rendered as `powerNotch of powerMax` (arithmetic, no index); the inline keymap (§2.3) is an object-literal lookup keyed by `KeyboardEvent.code` whose miss yields `undefined` and is filtered before building the intent, not an array index. No Phase-1 lookup site beyond `nextStop` indexes an array, so `nextStop` is the single guarded site. **Only its DOM rendering is the untested adapter** — `createHud` builds per-field `<span>`s once and patches `textContent`/lamp classes (amber for DSD warning + DRA set, red for penalty) on `update`. No jsdom dependency is incurred. The HUD is a sibling DOM overlay, not routed through `SceneHandle`.

### 2.10 What stays in the thin, untested `main.ts`

Only impure glue, no decision logic — explicitly the **deliberately-untested adapter surface**:
- create scene/route/spec/HUD and initial `SimState`/`ControlState`/`SafetyState`.
- `keydown` (`if (e.repeat) return`)/`keyup`/`blur` listeners; the per-frame edge set; the inline keymap lookup (§2.3).
- the rAF loop and **the only wall-clock:** `dt = min((now - last) / 1000, 0.05)`.
- per-tick orchestration (calls in order, no branching):

```
read edge set → intent = intentFromKeys(edgeSet)            // inline lookup
controls = reduceControls(controls, intent, state, safety)  // safety = prior frame's (read before tickSafety updates it) → one-frame lag, §2.4
safety   = tickSafety(safety, intent, state, dt, tickAws(state, dt))
inputs   = resolveInputs(controls, safety, ADHESION.wetNight)
state    = step(spec, route, state, inputs, dt)
scene.render(state.chainage)
hud.update(buildHudView(state, controls, safety, route))
clear edge set
```

`mu` stays hardwired `ADHESION.wetNight` (no env var, no flag). Every branch that could be wrong lives in a pure, tested function; `main.ts` contains no control/safety/HUD arithmetic.

**Tab-hidden `dt` note.** In production `tickSafety` receives the **same clamped** `dt = min((now − last)/1000, 0.05)` as `step()` — both are fed the one rAF `dt` (§2.10 wiring above), so a long stall (tab hidden, machine asleep) advances vigilance by **at most `MAX_DT` (0.05 s) per frame** and can never trip a penalty on tab-return: the timer cannot jump 60 s in a single re-entry. The **only** path that feeds `tickSafety` an un-clamped large `dt` is the unit test (O8a/O8b, which deliberately exercises coarse `dt` to pin frame-rate independence) — never production.

**Emergency decel — the new pure physics path**
- **O1 (analytic stop distance).** Strip Davis (`davisA/B/C = 0`); `v0 = 20`; `mu = ADHESION.dry` (so the brake is **not** adhesion-limited — see R3 caveat in §5); `{ notch:0, brake:1, dir:1, emergency:true, mu }`. **Start from a FULLY-APPLIED brake** — construct the start state `{ chainage:0, speed:v0, brakeActual:1, time:0 }` (exactly as the existing braking-distance oracle's start state at `physics.test.ts:117`) so the ~1.5 s brake build-up lag (`BRAKE_LAG.buildTau`) does **not** corrupt the closed-form match; from `brakeActual=0` the ±3% analytic tolerance is unreachable. Drive to rest: `end.speed < 1e-3` and `end.chainage ∈ [0.97, 1.03] × v0²/(2·brakeEmergencyDecel)`.
- **O2 (emergency strictly shorter than service).** Same setup — including the **fully-applied start state** `{ chainage:0, speed:v0, brakeActual:1, time:0 }` so neither run pays the build-up lag — `emergency:false` (full service) vs `emergency:true`: emergency stop distance `<` service stop distance, ratio ≈ `0.9/1.3` within tolerance. Proves the flag changes physics (guards against "faked as brake=1").
- **O21 (emergency static-hold STRICTLY beats service — a real differential test).** Pins D22's `acceleration()` static-hold property (§2.7) **non-vacuously**, on a **synthetic steep grade** chosen so the two decels straddle the gravity load, run at the signature adhesion `mu = ADHESION.wetNight` (μ=0.2). Mirroring `physics.test.ts`'s existing synthetic `flatRoute` idiom, build a one-segment route with `grade = 0.12`. The grade is picked so `mass·G·grade` lies **strictly between** `brakeServiceDecel·meff` and `brakeEmergencyDecel·meff`: with `mass = 160 000`, `G = 9.80665`, `meff = 160 000·1.08 = 172 800`, the service hold force is `0.9·172 800 ≈ 155 520 N`, the emergency hold force is `1.3·172 800 ≈ 224 640 N`, and the gravity load is `160 000·9.80665·0.12 ≈ 188 288 N` — so service **cannot** hold (155 520 < 188 288) while emergency **can** (224 640 > 188 288). **The μ=0.2 adhesion limit is not binding here:** at this fixture the brake adhesion limit is `mu·mass·G = 0.2·160 000·9.80665 ≈ 313 813 N`, which exceeds both hold forces, so neither service nor emergency is adhesion-clipped — the differential is driven purely by the `brakeEmergencyDecel`-vs-`brakeServiceDecel` selection, exactly the property under test. The differential assertion: at a stand, no power (`notch:0`), fully-applied brake (`brakeActual:1`), on that grade at μ=0.2, drive a few seconds from `{ speed:0, brakeActual:1 }` — **service (`emergency:false`) SLIPS** (`|end.speed|` grows from 0, strictly `> 0`) while **EMERGENCY (`emergency:true`) HOLDS** (`|end.speed| ≈ 0` within `1e-3`). This test would FAIL if `emergency` were dropped from `brakeForce`/the static-hold branch, so it pins D22 with no vacuity. *Secondary sanity check (on the real route):* on `WESTFORD_EASTBANK`'s actual steepest grade by magnitude (0.012 — far below the slip threshold ≈ 0.099) **both** service and emergency hold (`|speed| ≈ 0`), confirming the synthetic grade is the discriminating case, not the route.
- **Regression.** The existing `brakeServiceDecel` braking-distance oracle (`physics.test.ts:109`) and the RK4 oracle still pass with `emergency:false`.

**DSD / shared penalty latch — `tickSafety` (+ `resolveInputs` + `step`)**
- **O8a (frame-rate independence — END-STATE equality).** `tickSafety`'s `dt` is **not** clamped, so a single large-`dt` step can jump past both the warn and penalty thresholds in one tick. This half therefore asserts only **end-state equality**, not intermediate crossings. With the train **moving** (`|state.speed| > STAND_EPS`) and **no `vigilancePing`**, advance the **same total elapsed time = `DSD_PERIOD + a few seconds`** (enough to cross *both* the warn window and the penalty threshold) in two `dt` regimes — one coarse `dt` call vs many fine `dt` calls — with identical inputs (`state.speed`, `aws`, no ping) on every step. The two regimes must yield equal `vigilanceTimer` (within floating-point tolerance), equal `dsdWarning` (bool, **exact**), and equal `penaltyReasons` (**EXACT `Set` equality**). **The float tolerance applies to the numeric `vigilanceTimer` only.** This pins that timed safety integrates purely off `dt` and the end state is independent of how the frame budget is sliced (R4).
- **O8b (warn→penalty CROSSING — FINE-`dt` regime only).** Because the coarse-`dt` regime can step *past* both thresholds in one unobservable tick, the warn-then-penalty crossing is checked **only in the fine-`dt` regime**, where intermediate states are observable. Advancing fine `dt` steps (same moving, no-ping inputs): `dsdWarning` flips `true` exactly at the warn crossing (`vigilanceTimer ≤ DSD_WARN_WINDOW`) while `penaltyReasons` is still empty, and then `"DSD"` latches into `penaltyReasons` at the penalty crossing (`vigilanceTimer ≤ 0`) — catching a `<` vs `<=` / ordering / stateful-latch bug. (No "in both regimes" claim — the coarse regime cannot observe the crossing.)
- **O8 complementary static guard (the no-wall-clock half).** A grep over `src/sim` finds no `Date`/`Math.random`/`performance.now`/`setTimeout` (the no-wall-clock definition-of-done; there is no `src/input` directory — the keymap is inline in `main.ts`, whose rAF `dt` is the one allowed wall-clock, D10).
- **O9 (timeout → penalty → stop — train must be moving, on the SYNTHETIC `flatRoute`).** Drive the train **moving** (`|state.speed| > STAND_EPS`, since the vigilance countdown is motion-gated — §2.6 step 2) with no `vigilancePing` for `DSD_PERIOD` s: `dsdWarning` becomes true at the warn window, then `penaltyReasons` contains `"DSD"`. Companion check: a *parked* train (`|state.speed| ≤ STAND_EPS`) ticked for the same elapsed time holds its timer and **never** latches DSD. Penalty active ⇒ `resolveInputs` forces `brake=1, emergency=false, notch=0`; the moving-stop leg drives from `speed ≈ 20 m/s` on the **synthetic `flatRoute`** (the genuinely-level route used in `physics.test.ts:16` — `grade = 0` for its whole length, no stations) and reaches `|speed| ≈ 0`. Running on `flatRoute` (rather than a named `WESTFORD_EASTBANK` segment) removes the "must fit inside a short level window" constraint and makes the no-gradient-roll-back guarantee **exact** — there is no gradient anywhere on the route to confound the assertion.
- **O10 (ack at a stand releases the latch — the AVOID-an-un-releasable-latch oracle).** While stopped (`|state.speed| ≤ STAND_EPS`) and penalty-latched, an `acknowledge` intent empties `penaltyReasons`; afterwards `resolveInputs` with full power + no brake yields `notch === 1` (power available again). (The vigilance timer is left holding because the train is at a stand — §2.6 step 2 — so the countdown only resumes once it moves again; a `vigilancePing` resets it to `DSD_PERIOD` if the driver acts.)
- **O11 (ack while moving does NOT release).** Penalty latched, `state.speed = 5`, `acknowledge` ⇒ `penaltyReasons` still contains `"DSD"`.
- **O12 (penalty ≠ emergency decel, on the SYNTHETIC `flatRoute`).** Under penalty, `resolveInputs(...).emergency === false` and `brake === 1` (full service, not slam) — pins the brief's §5 semantics. The driving leg runs from `speed ≈ 20 m/s` on the **synthetic `flatRoute`** (`physics.test.ts:16`; `grade = 0` everywhere, no stations) so the full-service penalty brake brings the train to `|speed| ≈ 0` without any gradient roll-back confounding the assertion and with no short-level-window constraint.
- **O23 (penalty brake HOLDS on real graded track).** O9/O12 run on the synthetic `flatRoute`; this oracle pins that the full-service penalty brake genuinely holds on a gradient too. Under penalty (resolved demand = full SERVICE, `emergency:false`, `notch:0`) at a stand on `WESTFORD_EASTBANK`'s **representative falling ~1% grade — the Eastbank approach at chainage ~5 200** (route grade `−0.01`, e.g. start state `{ chainage:5_200, speed:0, brakeActual:1, time:0 }`) at `mu = ADHESION.wetNight`, drive several seconds: `|speed|` stays `≈ 0` (within the existing `0.05` tolerance the level-and-falling no-creep property uses) and chainage barely moves — the penalty brake holds on graded track, not only on the level. (The existing `physics.test.ts` "also holds on a falling gradient" test already proves a full brake holds at chainage 5 200; O23 pins the same property via the *penalty*-resolved demand.) Note: this is a representative falling grade, **not** the route's steepest grade by magnitude — that is the +1.2% Riverside climb (route grade `0.012`); the wording deliberately drops "steepest". (Distinct from O21, which is the *emergency* static-hold differential on a synthetic steep grade; O23 is the *service/penalty* hold on a real falling route grade.)
- **O18 (positive DSD — periodic pings never penalise).** A `SafetyState` that receives a `vigilancePing` strictly more often than every `(DSD_PERIOD − DSD_WARN_WINDOW)` seconds — i.e. with a ping interval `< 53 s` for the 60/7 constants, e.g. ping every **50 s** — while the train is **moving** (`|state.speed| > STAND_EPS`), keeps `dsdWarning` **false** AND `penaltyReasons` **empty** throughout a `2 × DSD_PERIOD` run. (The `< (DSD_PERIOD − DSD_WARN_WINDOW)` bound is what guarantees each ping resets the timer before it can even enter the warning window, so the warning lamp never lights and the latch never trips.) The positive complement of O9 — pins the intended deadman behaviour (§2.3/§2.6) from the safe side, so an attentive driver is never penalised.
- **O19 (EMERGENCY latches — release only at a stand).** A driver-commanded EMERGENCY (`brakeStep = BRAKE_EMERGENCY`) **cannot be released while moving:** with `|state.speed| > STAND_EPS`, a `brakeDown` intent leaves `brakeStep === BRAKE_EMERGENCY` (unchanged). Once `|state.speed| ≤ STAND_EPS`, the same `brakeDown` decrements to `BRAKE_FULL (3)`. (Pins D18; complements O17, which covers the service-step decrements that are always free.)
- **O22 (power handle returns to OFF after a penalty — no lurch on release).** Pins D23, the power-to-OFF-under-penalty interlock (§2.4 step 2). (a) **Latch forces OFF:** start with `powerNotch > 0` (e.g. 4); pass a `safety` with `penaltyReasons ∋ "DSD"` into `reduceControls(c, intent, state, safety)` ⇒ the result has `powerNotch === 0`. (b) **No instant restore on release:** after the penalty is released (ack at a stand, `penaltyReasons` empty), with the train at a stand, a frame with **no `powerUp` edge** leaves `powerNotch === 0` (releasing the penalty did not restore the pre-penalty notch); a `powerUp` edge **while the penalty is still active** is overridden back to `0`; only a fresh `powerUp` edge on a frame where `penaltyActive(safety)` is false advances `powerNotch` to `1`. So power resumes solely by a deliberate fresh demand from OFF — never by the latch clearing.

**Simultaneous / conflicting edges — `reduceControls`**
- **O20 (one-frame conflict resolution).** Pins §2.3 ordering: an opposing `powerUp`+`powerDown` in the same intent ⇒ `powerNotch` unchanged; an opposing `brakeUp`+`brakeDown` ⇒ `brakeStep` unchanged; an `emergency` edge together with `brakeDown` ⇒ `brakeStep === BRAKE_EMERGENCY` (emergency overrides the brake edges); two reverser edges in one frame (e.g. `reverserFwd`+`reverserRev`) ⇒ `reverser`/`lastDir` unchanged. **HUD projection — `buildHudView` (pure)**

  - **SPAWN (boundary — station at exact current chainage is already reached):** at `chainage = 0` with `lastDir = +1`, `nextStop === "Riverside"` (chainage 1 500) — **NOT** "Westford", which sits at chainage 0 and counts as already reached, not the next stop (§2.9 boundary rule: a station is a candidate only when `(station.chainage − chainage)·lastDir > 0`).

**Power-inhibit isolation — `resolveInputs` (one inhibitor at a time)**

These pin each OR-term of the power-inhibit composition (§2.5 step 3) *independently*, so no single inhibitor can silently stop working behind another. Baseline fixture = all-clear: `reverser:"FWD"`, `brakeStep:0`, `dra:false`, no penalty, `powerNotch:POWER_NOTCHES`. Each asserts `resolveInputs(c, s, mu).notch`:

- **O15a (brake-only inhibits).** From the baseline, set `brakeStep:1` (only) ⇒ `resolveInputs(c, s, mu).notch === 0`.
- **O15b (reverser-OFF-only inhibits).** From the baseline, set `reverser:"OFF"` (only) ⇒ `resolveInputs(c, s, mu).notch === 0`.

## 5. Residual risks

1. **EMERGENCY adhesion caveat (keep — not a bug).** EMERGENCY stays distinct from FULL SERVICE *only while not adhesion-limited*. At `ADHESION.wetNight` (μ=0.2) the adhesion limit is `0.2·160000·9.80665/meff ≈ 1.81 m/s²`, which exceeds 1.3, so emergency is **not** clipped at the signature setting — but at very low grip the limit drops below 1.3 and emergency ≈ service. This is physically correct, not a defect; a future reader must not "fix" it. The analytic oracle (O1/O2) therefore uses `ADHESION.dry` (limit ≈ 2.72 m/s² > 1.3), exactly as the existing braking-distance oracle does, so the `v²/(2·1.3)` check stays valid.
2. **Penalty-release ordering (the load-bearing seam).** Phase 1's release rule — "clear on an ack edge AND at a stand" — is simple, but the *empty-set-clears-latch* invariant and the Phase-2 extension (reasons that persist after ack must fail a "source no longer demands it" predicate, kept as a forward-looking note in §2.6) are the one place a future bug could strand the player or make the latch un-releasable. Mitigated by O10/O11 and the documented contract in `controls.ts`; the forward note names the persisting-reason case explicitly so Phase-2 AWS slots in without touching the release machine.
3. **`exactOptionalPropertyTypes` friction.** Making `emergency` a required field ripples into existing `AccelInputs`/`SimInputs` literals (physics test + `main`). Mitigated: it's a required boolean with an obvious `false`; the impact map lists every literal to touch — small and mechanical.
4. **Untested adapter surface** (inline keymap, edge-set construction, rAF, DOM writer) can hide wiring bugs no oracle catches — e.g. not clearing the edge set per frame → double-stepping. Mitigated by keeping it to a few lines (`if (e.repeat) return` + clear-per-frame is the whole logic) and pushing every decision into pure functions; the residual is intentional and named. Manual smoke-test via `vite dev`.
5. **Shared epsilon / feel tuning.** `STAND_EPS = 0.3` is reused for reverser, DRA, EMERGENCY release, and latch release — coherent but tuning one tunes all; `DSD_PERIOD`/`DSD_WARN_WINDOW` are guesses, and linear `notch/Nmax` gives even effort steps that may feel un-prototypical against a real motor curve (the constant-power taper in `tractiveEffort` softens this). All are single-point named constants; oracles assert behaviour *relative to* the constants, not magic numbers, so tuning won't break tests. None is a blocker.
6. **Phase-2 coupling debt.** The latch's `Set<PenaltyReason>` and the `tickAws` seam are designed for AWS/TPWS but unexercised in Phase 1; O13 is the only guard that the seam stays shaped right. Flagged as the PLN's "on-ramp" — kept minimal, not over-built.
