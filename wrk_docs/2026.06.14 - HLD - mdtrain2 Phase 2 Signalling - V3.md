## 1. Scope and non-goals

### In scope

### 2.2 Aspect rule (pure, train-free)

A starter `i` is **held RED** iff its `protects` station is **not in `served`**. A plain block signal (`protects === ""`) is never self-held. Otherwise it cascades off **one** signal ahead. `served` lives in `AwsState` (Â§2.3). Test own held-red FIRST, then look ahead:

```ts
// route.ts â€” pure, O(1) per signal amortised, no train, no dt
export function aspectAt(route: Route, i: number, served: ReadonlySet<string>): Aspect {
  const sig = route.signals[i];
  if (!sig) return "GREEN";                                   // guard (noUncheckedIndexedAccess)
  if (sig.protects !== "" && !served.has(sig.protects)) return "RED"; // OWN held-red first
  const ahead = route.signals[i + 1];                         // next signal toward Eastbank
  if (!ahead) return "GREEN";                                 // last signal, nothing ahead
  const aheadAspect = aspectAt(route, i + 1, served);         // backward sweep, depth â‰¤ n
  if (aheadAspect === "RED") return "YELLOW";
  if (aheadAspect === "YELLOW") return "DOUBLE_YELLOW";
  return "GREEN";
}
```

Recursion depth bounded by `signals.length`; index strictly increases â‡’ no cycles; terminates at the last signal (A4 pins the guarded out-of-range read). A plain block signal between two reds correctly shows the intermediate caution and never blocks the cascade. Double-yellow is reachable: with only S4 unserved and the block signals GREEN, the nearest-ahead-red lookback through S3/S2 still yields S3=YELLOW, S2=DOUBLE_YELLOW (the block signals between them participate in the chain as transparent GREEN-or-cascade links).

**Served-set writer.** `served` grows in the `tickAws` path: a station enters `served` the first tick the front is within `Â±platformHalf` of its stop board **and** `|speed| â‰¤ STAND_EPS`. The only writer; position-only proxy, not a dwell model (Residual #2); **monotonic add-only** (F7); the one `dt`-sensitive write (G-note, Â§4 G; F12).

### 2.3 `AwsState` + the new `tickAws` signature and state machine

`prevChainage` is passed **explicitly** (not folded into `AwsState`); `dir: 1 | -1` is passed **explicitly** (F5): at a stand `sign(state.speed)` is undefined, so heading comes from the same authority `resolveInputs` uses, `ControlState.lastDir`. `main.ts` derives `dir` from `controls.lastDir`; `tickAws` stays pure on its explicit arguments.

```ts

export type AwsPhase = "CLEAR" | "WARNING" | "BRAKE";
export type Sunflower = "BLACK" | "CAUTION"; // all-black = clear; black/yellow = caution-latched

export interface AwsState {
  phase: AwsPhase;        // AWS warning/brake machine
  warnTimer: number;      // s remaining in the ack window; counts DOWN off dt (WARNING only)
  sunflower: Sunflower;   // latched caution reminder; reset only by a GREEN magnet
  brakeReason: PenaltyReason | null; // single BRAKE-latch source: "AWS" | "TPWS" | null
  served: ReadonlySet<string>;       // stations whose booked stop is done (occupancy proxy)
  spad: boolean;          // sticky SPAD event flag (HUD/scoring)
}

export interface AwsHud { sunflower: Sunflower; spad: boolean; } // F8/F11: NO aspect field here
export interface AwsOutput { reasons: PenaltyReason[]; }         // UNCHANGED latch-boundary type

export function createInitialAws(): AwsState {
  return { phase: "CLEAR", warnTimer: 0, sunflower: "BLACK", brakeReason: null,
           served: new Set(), spad: false };
}

export function tickAws(
  aws: AwsState,
  state: SimState,
  route: Route,
  intent: ControlIntent,
  prevChainage: number,
  dir: 1 | -1,          // F5: explicit heading (from ControlState.lastDir in main.ts)
  dt: number,
): { next: AwsState; reasons: PenaltyReason[]; hud: AwsHud };
```

**No `releaseArmed`, no one-tick lag (V3, F8).** The DSD-scoped carry-forward seed (P2-RA) already makes AWS/TPWS releasable. The genuine "survive ack" requirement is **ack-WHILE-MOVING**, not "survive a stand-ack while still demanding." So an ack-at-stand in BRAKE clears the reason **the same tick**; an ack-while-moving never clears it. This removes a whole field and a whole transition â€” simpler to hold in the head.

**One BRAKE latch path for AWS + TSS + OSS.** A single `phase === "BRAKE"` plus `brakeReason: "AWS" | "TPWS" | null` carries the latched demand. Both an AWS-deadline miss and a TPWS TSS/OSS trip set `phase = "BRAKE"` and `brakeReason`; both re-assert their own reason every BRAKE tick and both clear identically (D5).

#### The state machine (one tick, all `dt`-counted)

Front position uses fixed `FRONT_OFFSET` (legacy 2.0 m) **in travel direction**: `frontNow = state.chainage + dirÂ·FRONT_OFFSET`, `frontPrev = prevChainage + dirÂ·FRONT_OFFSET`. **All edge logic fires only on strict forward motion `frontNow > frontPrev`** â€” reverse/standing crosses nothing (D10).

1. **Served update**: if `|speed| â‰¤ STAND_EPS` and front within `Â±platformHalf` of a station, add it to `served` (copy-on-write, add-only). Clears that station's starter (relaxes the cascade behind it).
2. **Magnet crossings** (`pointsCrossedFwd` over postsâˆ’`AWS_MAGNET_OFFSET`): for each magnet crossed, evaluate that signal's `aspectAt` over the (possibly just-updated) `served`:
   - **GREEN magnet** (a block signal, or a served station's starter) â†’ `sunflower = "BLACK"` (clear bell); if `phase === "WARNING"`, cancel to `CLEAR`. Does **not** clear a `BRAKE` (F6: guarded on `phase !== "BRAKE"`).
   - **non-GREEN magnet** â†’ only if `phase !== "BRAKE"` (F6): `phase = "WARNING"`, `warnTimer = AWS_WARN_WINDOW`, `sunflower = "CAUTION"`. A latched BRAKE outranks a fresh magnet warning.
3. **WARNING countdown**: if `phase === "WARNING"`:
   - `intent.acknowledge` (allowed **while moving** â€” D4) â‡’ `phase = "CLEAR"`, sunflower stays `CAUTION` (latched reminder), no penalty.
   - else `warnTimer âˆ’= dt`; if `warnTimer â‰¤ 0` â‡’ `phase = "BRAKE"`, `brakeReason = "AWS"`.
4. **TPWS / SPAD** (`pointsCrossedFwd` over the signal-post array, evaluating `aspectAt` at the crossing):
   - **TSS / SPAD**: a starter crossed while `aspect === "RED"` (station **unserved** â€” a blow-through) â‡’ `phase = "BRAKE"`, `brakeReason = "TPWS"`, `spad = true`. A served station's starter is GREEN, and a block signal is never RED on a single-train route, so a normal stop crosses nothing red â€” SPAD-free.
   - **OSS**: the signal's OSS loop crossed while that signal `aspect === "RED"` **and** `|speed| > OSS_TRIP_SPEED` â‡’ `phase = "BRAKE"`, `brakeReason = "TPWS"`.
   - **Within-tick precedence (F6/F12)**: when a single coarse `dt` sweeps past *both* a magnet (step 2) and its red starter (step 4), AWS is evaluated first (may fall to BRAKE on `brakeReason="AWS"`), then TPWS/TSS **overwrites `brakeReason` to `"TPWS"`** and sets `spad`. Deterministic order: *served â†’ magnet/AWS â†’ TPWS/SPAD â†’ release*. TSS/SPAD is authoritative (the train physically passed the red), so `"TPWS"` wins; the latch behaviour (full-service brake) is identical either way.
5. **BRAKE clearing â€” NO LAG (V3, F8).** While `phase === "BRAKE"`:
   - If `intent.acknowledge && |speed| â‰¤ STAND_EPS` (**ack at a stand**): transition `phase â†’ CLEAR`, `brakeReason â†’ null` **this same tick**; sunflower stays `CAUTION` until the next GREEN magnet. Step 6 then returns `[]`, so the DSD-scoped seed carries nothing forward and the latch releases the same tick (C2/C2-PIN).
   - Else (ack-while-moving, or no qualifying ack): **re-assert** `brakeReason`, stay in BRAKE. Step 6 returns `[brakeReason]`; the fold re-adds it, so it **survives** (C4). This is the real "survive ack" case.
   - (`spad` stays sticky â€” a scored event.)
6. **Reasons assembled**: `reasons = brakeReason ? [brakeReason] : []`.
7. **HUD**: return `{ sunflower, spad }` **only** (F8/F11). The HUD aspect is **not** computed here; `buildHudView` derives it from `served` via `nextSignalAhead` + `aspectAt` (Â§2.6).

All countdowns are `dt`-subtractive and all crossings use swept intervals, so coarse and fine `dt` reach equal end-states for everything **except** the served-set sample (the one `dt`-sensitive write â€” G-note, F12). A coarse `dt` jumping a magnet *and* its starter in one tick is handled because steps 2 and 4 consult the same swept interval, in the F6/F12 order.

**Ack split (D4 / F9).** The WARNING-window ack (cancel-before-penalty) is internal to `tickAws` and allowed while moving; the latch-release ack (clearing an *asserted* BRAKE reason) is ack-at-stand. **Both â€” plus the DSD release in `tickSafety` â€” consume the single `intent.acknowledge` edge.** One driver acknowledge that cancels the AWS warning, releases a stand-still penalty, and resets DSD is **intended and realistic**. No separate ack path.

### 2.4 How reasons survive / clear with no lag (load-bearing)

This is the keystone mechanism. It depends on **two** facts in `tickSafety`: the carry-forward seed at `controls.ts:258` is **scoped to DSD-only** (P2-RA), and the fold at `:275` runs **before** the DSD-only delete at `:278-281`.

**The seed scoping (the one-line edit at `:258`):**

```ts

if (s.penaltyReasons.has("DSD")) reasons.add("DSD");
```

This replaces the original `const reasons = new Set<PenaltyReason>(s.penaltyReasons);`. After this edit, AWS/TPWS reasons are **never seed-carried** â€” re-derived **fresh from the fold** (`for (const r of aws.reasons) reasons.add(r)`, `:275`) every tick. Present in the set **iff `tickAws` returns them this tick**.

- **Survives ack-WHILE-MOVING (C4).** While `phase === "BRAKE"` and the driver acks at speed, `tickAws` keeps returning `[brakeReason]` (step 5's else branch re-asserts). The fold (`:275`) re-adds it; the DSD-only delete at `:278-281` cannot touch it. So an ack-while-moving demanding a brake does **not** strand-clear AWS/TPWS â€” the source still demands it, exactly the pre-authorised clause (`controls.ts:244-246`). This is the genuine "survive ack" case.
- **Clears on ack-AT-STAND the SAME tick (C2).** On an ack-at-stand in BRAKE, `tickAws` transitions `phase â†’ CLEAR`, `brakeReason â†’ null`, and returns `[]` **that same tick** (no lag). The fold adds nothing; **because the seed is DSD-scoped** the reason is not seed-carried, the set empties â†’ the latch releases the same tick. **This requires the `:258` seed-scoping**: under the original full carry-forward seed the reason would re-seed every subsequent tick from `s.penaltyReasons` and never be deleted (the release names only "DSD"), making the latch un-releasable forever.
- **DSD coexistence (C-COEXIST).** DSD + an AWS/TPWS reason both latched: an ack-at-stand deletes `"DSD"` (release machine) *and* `tickAws` stops returning AWS/TPWS the same tick (fold adds nothing) â‡’ **both clear together** under the single acknowledge â€” acceptable, one driver acknowledgment.
- **Why the release machine stays untouched.** The `:280` comment ("Phase-2 persisting reasons would re-add themselves above") assumed fold-sourced reasons are **not** seed-carried. The seed-scoping makes that true: every non-DSD reason in the set is, by construction, this-tick fold output, and drops automatically when the fold stops producing it. So the release machine (`:277-281`), `resolveInputs`, `reduceControls` need **no edit** â€” only the seed is scoped.
- **`tickAws` only ever touches its own reasons** â€” never `"DSD"` (R1), and DSD logic never touches `"AWS"`/`"TPWS"`.

> **`tickSafety` release machine + signature unchanged; the carry-forward seed scoped to DSD-only (one line, `:258`).** Survive (ack-while-moving) = the fold re-adds the reason every tick `tickAws` still demands it; clear (ack-at-stand) = `tickAws` stops demanding it the same tick **and** the DSD-scoped seed no longer carries it â‡’ it drops. No `releaseArmed`, no lag.

### 2.5 `tickSafety` + frame order + test-helper ripple

- **`tickSafety` release machine + signature are UNCHANGED; the carry-forward seed is scoped to DSD-only (one line, `:258`).** It still takes `aws: AwsOutput` and folds `aws.reasons` at `:275`. ## 5. Residual risks

1. **Fold-before-delete order + seed-scoping coupling (highest).** Persistence/release rests on two coupled facts: the seed at `:258` is **DSD-scoped**, and the fold (`:275`) runs **before** the DSD-only delete (`:278-281`). The un-releasable-latch risk is resolved by the seed-scoping; the survive-ack risk is resolved by `tickAws` re-asserting while moving (no lag needed). What remains fragile is **fold-order**: moving the fold after the delete, or generalising the delete to non-DSD reasons, would mis-clear mid-demand. C2/C2-PIN/C4/C-COEXIST are the guards; comments cite `:258`, `:275`, `:278-281`.
2. **Served-set as occupancy proxy is a fiction, and is `dt`-sensitive.** Aspect colour couples to "drove well enough to stop." The exact served-flip tick depends on `dt` slicing (G-NOTE) â€” pinned, defined, excluded from the dt-independence claim. A train crawling through a platform without truly stopping could mis-flag. If gamey, swap `served` for a scripted held-signal list with zero cascade changes.
3. **Double-yellow only on the S1â†’S2 section.** A1/A-BLOCK prove the logic; the route's *use* of YY is decorative-but-correct.
4. **OSS abstraction.** Speed-threshold-at-a-loop-chainage, not real loop-spacing/arm-timer physics. Trip speed is one tunable constant.
5. **OSS loop placement (F9).** Each OSS loop is `post âˆ’ OSS_LOOP_OFFSET(=50)`, sitting just behind its starter, so it is reached (and can arm) on the approach **before** the starter it protects. All four OSS loops (1570, 3170, 4470, 5950) lie ahead of their derived magnets and behind their posts â€” correct arm-then-trip ordering. (Block signals carry no OSS loop.)
6. **Direction reconciliation.** Heading comes from explicit `dir = ControlState.lastDir` (F5), not `sign(speed)`. All crossings gate on strict `frontNow > frontPrev`; D2/G2 guard reverse and coarse-dt.
7. **`prevChainage` + `dir` + frame-reorder plumbing is in the impure shell.** A wiring bug (post-step chainage as prev, wrong `dir`, or not capturing prev before `step()`) is untestable at `main.ts`. Mitigation: `tickAws` is pure on its explicit `prevChainage`/`dir`, so the *logic* is fully tested (G1 constructs the pair directly); only the plumbing is trust-based.
8. **Starter geometry (F1, resolved).** `STARTER_OFFSET=120 > max(platformHalf)=100 + FRONT_OFFSET=2`, so a normal stop anywhere in-platform serves before the front reaches the RED post. D1-SERVED and D-OVERSHOOT pin it. The V2 `STARTER_OFFSET=60` SPAD-while-moving bug is fixed.
9. **F7 reverse-and-re-approach (out of scope).** `served` is monotonic add-only.
