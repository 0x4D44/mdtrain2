# 2026.06.14 — PLN — mdtrain2 design

## Purpose

A new, improved train simulator succeeding `mdtrain` ("UK EMU Cab Simulator").
The original is preserved unchanged (the `0x4D44/mdtrain` repo, and `legacy/`
here) as a record of what AI coding tools produced in early 2026. mdtrain2 is
the "what they can do now" answer.

## Agreed brief

- **Primary lens:** immersive experience — but built on an engineered, tested
  core (presentation rides on correctness, not the other way round).
- **The signature scene:** a GTO-inverter EMU running an improved
  Westford→Eastbank line, **at night in the rain** — wipers, signal glow,
  wet-rail reflections — driven first-person from a detailed cab, with the
  inverter "gearbox" whine rising under power.
- **Tech:** Vite + TypeScript. A pure, deterministic **sim core** (no Three.js)
  separated from rendering, audio, input and UI. Vitest test oracles over the
  core. Builds to static files (hostable on Pages like `worldviewer`).
- **Scope discipline:** one route segment done beautifully, not a long line.

## Architecture

```
src/
  sim/        pure TS, deterministic, tested — the single source of truth
    physics.ts     traction-effort & Davis resistance (A+Bv+Cv²), gradient,
                   brake build-up/release, adhesion; fixed-timestep integrator
    train.ts       EMU spec (mass, power, brake rates, motor curve) + state
    route.ts       chainage, curvature, gradient, cant, stations, PSR/TSR
    signalling.ts  block/aspect state machine, AWS, TPWS, SPAD     (later)
    timetable.ts / scoring.ts                                       (later)
    simulation.ts  one tick: (state, inputs, dt) -> next state
  render/     Three.js: cab, world, instruments, camera (reads sim state)
  audio/      sample beds + procedural GTO inverter synthesis        (later)
  input/      keyboard / touch / gamepad -> control inputs
  ui/         HUD, menus
test/         vitest — physics, signalling sequences, braking distance
legacy/       the original mdtrain, kept runnable for before/after
```

### Test oracles (the multiplier)

- **Differential:** the production semi-implicit integrator vs an independent
  RK4 of the same ODE agree within tolerance over coast and braking runs.
- **Analytic:** braking distance ≈ v²/(2·decel) where resistance is small.
- **Properties (regressions from the original's bugs, now un-regressable):**
  - doors open + brake applied ⇒ train never moves (no creep);
  - forward power moves +chainage, reverse moves −chainage (reverser sane);
  - AWS unacknowledged in N s ⇒ always reaches BRAKE;
  - acknowledge ⇒ the AWS brake latch always releases.

## Modelling choices (Phase 0, all tunable)

- **Unit:** ~4-car GTO EMU (Class 365-ish). Loaded mass 160 t; rotational-inertia
  factor 1.08; P_max 1.0 MW; TE_max 120 kN; v_max 100 mph (44.7 m/s).
- **Resistance:** Davis R(v)=A+Bv+Cv², A≈3000 N, B≈120, C≈7.
- **Adhesion:** μ≈0.30 dry, ~0.20 wet-night; caps tractive (and brake) effort.
- **Brake:** demand→actual via first-order lag (build-up ~1.5 s, release ~2 s);
  full-service ≈ 0.9 m/s², emergency ≈ 1.3 m/s².

## Phasing (each phase independently shippable)

0. **Foundation** — Vite+TS+Vitest scaffold, tested sim core, thin render thread
   that moves the camera along the rails end-to-end. ← *this phase*
1. **Driving** — cab controls (power/brake/reverser/DSD), HUD, drive the route.
2. **Signalling** — multi-aspect blocks, AWS/TPWS, speed profile, SPAD; tested.
3. **Presentation** — detailed cab (desk, sunflower, DSD), lighting, GTO audio.
4. **World** — route richness, station layouts, scenery, the wet-night weather.
5. **Reach** — touch/mobile, accessibility, performance pass.
