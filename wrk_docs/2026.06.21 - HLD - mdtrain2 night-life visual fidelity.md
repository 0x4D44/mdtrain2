# HLD — mdtrain2 night-life visual fidelity

**Status: DRAFT** · Author: design proof (Claude) · Date 2026-06-21 · Target `C:\language\mdtrain2`
· `package.json` 0.13.0 → **0.14.0**.

Raise lineside *perceived realism* for the night ride — lit-window buildings, UK colour-light
signal heads, a moonlit hero truss bridge, moon/stars, denser trackside lighting, and optional
road traffic — **with zero new npm deps, zero committed image assets, and no change to the pure
core**. Same quality brief as the 06-17 pass: simplicity, no overengineering, design for
testability, preserve the pure-core (`src/sim/*`) / impure-projector (`src/render/*`) split.

A working visual proof exists (`Night Cab Fidelity Proof.dc.html`) — it is the **source of truth
for look and technique**; this HLD maps it onto the real signatures.

## 1. Scope
**In:** lit-window facades (`textures.ts` + `scenery.ts`); UK signal-head **geometry** rebuild
(`scene.ts`); a procedural steel through-truss bridge + cool rim lights (`scenery.ts`); a
celestial layer — moon disc, bloom halo, star field — folded into the env lifecycle (`scene.ts`);
denser trackside lamps + warm ballast marker lights (`scenery.ts`); **optional** deterministic
road traffic with headlights (`scenery.ts` + the existing per-frame loop in `scene.ts`).

**Out (non-goals):** any new npm dep, env var, or feature flag; per-building unique textures
(see DL1); free-flying `Math.random` in the frame loop (DL3); new signalling *logic* (aspects
stay driven by the pure core — geometry only, per 06-17 DL4); imported glTF/Blender assets (a
later, separate decision — this pass is 100% procedural); water/weather changes.

## 2. Approach (by feature, minimal clean fix)

**A. Lit-window facades (buildings).** Today `buildBuildings` paints one `InstancedMesh` with a
masonry albedo. Add a `buildFacade()` producer to the texture layer (same Canvas-2D + deterministic
hash as `buildGround`/`buildMasonry`, SRGB-pinned per D23) that emits an **albedo + a matching
emissive map**: a window grid with ~half the cells lit warm. Wire `emissiveMap` + `emissive:
0xffffff` (+ modest `emissiveIntensity`) onto the buildings material. At night the existing bloom
turns dead boxes into a lit skyline; identical massing, so it's a pure-render upgrade. (Instancing
constraint → DL1.)

**B. UK signal heads (geometry only).** `buildSignals` **already** exposes `red / amberTop /
amberBot / green` and the pure signalling core already sets their `emissiveIntensity` (scene.ts
384–387) — keep that wiring untouched. Rebuild only the head **geometry** to UK pattern: aspects
stacked **vertically with RED at the BOTTOM**, each lamp recessed in a short black hood, on a black
backboard facing the track. No new state, no logic change (06-17 DL4 preserved).

**C. Moonlit truss bridge.** Add a `buildTrussBridge()` sibling to `buildOverbridge`, placed
identically (`placeOnCentreline(route, s, d)` for pose, `formationHeight(route, s)` for deck
clearance). Through-truss as cylinder chords/verticals/diagonals via a small `bar(a,b,r)` helper;
**high-metalness / low-roughness steel with `envMapIntensity` up** so it catches the env reflection,
plus cool emissive **navigation lights at each top-chord node** that rim-light the lattice against
the sky. Use as the hero overbridge at chosen chainages; box overbridge stays for ordinary ones.

**D. Celestial layer (env lifecycle).** Add a moon disc (`MeshBasicMaterial`, `fog:false`), a soft
additive halo sprite, and a star-field `Points` (`fog:false`). Position from the **env direction**
so it tracks day→sunset→night and **rebuilds/disposes on env change**, exactly like the existing
env equirect — i.e. it lives in the env setup, not a one-shot scenery build (DL2).

**E. Denser trackside lighting.** More lamp posts (warm emissive head + a hard-capped few
`PointLight`s) and a run of instanced warm **ballast marker lights**, all placed via
`placeOnCentreline`/`anchorY`, built once, zero per-frame allocation.

**F. Road traffic (optional).** Cars (dark body + emissive white headlights / red tail-lights) on
roads alongside the line and across the truss deck. Either **parked** (lit, cheapest, fully on
the build-once rule) or **animated deterministically** from the same per-frame loop in `scene.ts`
that already steps signal emissives — precomputed paths, no `Math.random` in-loop (DL3). Headlight
`PointLight`s hard-capped per quality tier (DL4).

## 4. Acceptance criteria (screenshot-gated unless noted)
- **AC-A** At night, lineside buildings show **lit windows** that bloom; daytime facades read as
  ordinary masonry (emissive washed out by exposure). Same massing as today.
- **AC-B** Signal heads are **UK pattern**: vertical stack, **red at the bottom**, hooded lamps,
  black backboard facing the track; the lit aspect still matches the pure signalling core (the
  existing aspect oracle/test stays green — **not** screenshot-only).
- **AC-C** The truss bridge reads as **lit steel** (catches the moon + node rim-lights), not a flat
  silhouette; placed correctly over the line via the centreline/formation helpers.
- **AC-D** A **moon + stars** are visible; they track the env (present at night, washed at day) and
  **dispose cleanly on env change** (no leak — pinned by an env-swap smoke check).
- **AC-E** Denser trackside lamps + ballast marker lights are visible and sit on the ground
  (`anchorY`), no float.
- **AC-F (optional)** Cars show headlights/tail-lights; if animated, motion is **deterministic**
  (no `Math.random` in the frame loop) and headlight `PointLight` count ≤ the tier cap.
- **REG-perf** Mobile tier still renders within budget (draw calls + `PointLight` count checked —
  see R1); REG-vis from 06-17 (viaduct/bore) still correct.

## 5. Decision log
- **DL1 Lit windows vs instancing.** Buildings are one `InstancedMesh` (one material) ⇒ instances
  can't each carry a unique emissiveMap. Ship a **shared** window emissive map on the existing
  InstancedMesh (one draw call); add variety only as **3–4 InstancedMeshes** gated by
  `quality.ts`. Per-building `Mesh`es are rejected on the desktop path only (R1/06-17 R3).
- **DL2 Moon belongs to the env, not scenery.** It tracks day/night and must rebuild/dispose with
  the env, so it lives in the env lifecycle — not the build-once scenery pass.
- **DL3 Traffic determinism.** If animated, paths are precomputed and stepped from the existing
  per-frame loop; **no `Math.random` in-loop** (keeps screenshots/oracle stable, honours the
  zero-per-frame-allocation rule for the static parts).
- **DL4 PointLight budget.** Emissive-only by default; real `PointLight`s (lamp pools, headlights)
  are hard-capped per tier — they are the only real cost here.
- **DL5 Signals geometry-only.** No block signals, no logic change; reuses the 4-aspect head the
  core already drives (06-17 DL4 carried forward).
- **DL6 100% procedural this pass.** No glTF/Blender import — that's a separate later decision for a
  small set of hero set-pieces; nothing in this HLD depends on it.

## 6. Residual risks
- **R1 Perf** — extra emissive geometry + lamps + (optional) cars + 3rd–5th InstancedMesh. Absorbed
  by instancing + tier gating; **re-check the mobile path** (draw calls, `PointLight` count). The
  hard cost is `PointLight`s → DL4.
- **R2 Lit-window uniformity** — a shared emissive map repeats the same window pattern on every
  block. Acceptable at night/at speed; 3–4 variants if it reads as tiling (DL1).
- **R3 Env-swap leak** — celestial meshes/sprites/points must dispose on env change (AC-D).
- **R4 Bridge placement** — truss must clear the loading gauge over the line; verify against
  `formationHeight` at its chainages (screenshot + the existing clearance check).
- **R5 Taste** — final look of facades / signal heads / truss / moon needs human eyes.

## 7. Suggested implementation order (vertical slices, cheapest-impact first)
1. Lit-window facades (`textures.ts` + `buildBuildings`). 2. UK signal-head geometry (`scene.ts`).
3. Celestial layer + denser lamps (`scene.ts` env + `scenery.ts`). 4. Truss bridge (`scenery.ts`).
5. Traffic (optional, `scenery.ts` + per-frame step). Each slice is one screenshot gate (TEST-7
style); the only new pure surface (an optional `celestialDir`, a `quality.ts` tier field) is
unit-tested.
