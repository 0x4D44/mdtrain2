## 1. Scope & non-goals

### In scope (control + HUD layer only)
- A **pure controls/safety reducer** (`src/sim/controls.ts`) holding `ControlState` + `SafetyState`, exposing the testable functions `reduceControls`, `tickSafety`, `resolveInputs`, and `buildHudView` that turn driver intent + timed safety into the existing `SimInputs` for `step()`.
- **Prototypical stepped controls:** notched POWER (OFF + 4 notches), stepped BRAKE (RELEASE + 3 service steps + FULL + EMERGENCY past an end gate), REVERSER (FORWARD/OFF/REVERSE).
- **Interlocks:** no power against an applied brake; reverser changes only at a stand; reverser OFF inhibits power.
- **Safety now:** DSD/vigilance (timer flavour) + DRA, feeding **one shared penalty latch + one acknowledge**.
- **One surgical pure-physics change:** a real, distinct EMERGENCY deceleration (`brakeEmergencyDecel=1.3`) selected in `brakeForce`, with its own analytic oracle — **not** faked as `brake=1`.
- A **keymap→intent input layer** kept **inline in `main.ts`** (decision-free ~20-line lookup; no separate module).
- A **structured DOM HUD** split into a pure `buildHudView` (tested) + a thin `createHud` DOM writer (untested).
- A **typed AWS/TPWS no-op stub** at `src/sim/safety/aws.ts` defining the Phase-2 seam and passing through unchanged.
- A thin, deliberately-untested `main.ts` wiring keyboard → controls → safety → `resolveInputs` → `step` → `render` → HUD.

### Non-goals (explicit — do not build)
- 3D cab interior, lighting, GTO audio (Phase 3); `scene.ts`, `index.html` 3D world unchanged.
- AWS/TPWS *behaviour* — magnets, deadlines, sunflower, SPAD (Phase 2). Phase 1 ships only the seam + no-op stub.
- Gamepad/touch input (Phase 5); keyboard only.
- Route signal/AWS-magnet positions, station-stop scoring (no data; later phases).
- Pneumatic pipe-propagation, per-axle brake modelling, continuous (analog) levers.
- Widening `SimInputs.dir` to include neutral `0` (breaking — `dir` stays `1|-1`; neutral lives only in the control layer).
- New state on `SimState`; latches/timers inside `step()`/`physics.ts`.
- Exposing `mu` to the driver — stays hardwired `ADHESION.wetNight`. No env vars, no feature flags, no CI changes, no new runtime deps.

## 2. The design

### 2.0 Risk-first overview — the five failure modes, designed away

This table leads because the rest of the design exists to defeat these. Each row maps a failure mode to the single design move that prevents it; the oracle that pins it is in §4.

| # | Failure mode | How it bites | Design move that prevents it | Oracle |
|---|---|---|---|---|
| **R1** | **Shared latch fights itself** (DSD now, AWS/TPWS later) | Two systems each "own" a latch; one clears it while the other still demands brake; or ack clears DSD but AWS re-latches next tick → un-releasable or prematurely-released brake | **One latch = `Set<PenaltyReason>`, active iff non-empty.** Each subsystem only adds/removes *its own* reason. Documented per-reason clearability predicate (§2.6). Ack is edge- and stand-gated. | O9–O12 |
| **R2** | **Interlock composition order** | "No power vs brake", DRA, penalty, reverser-OFF all mutate `notch`/`brake`; wrong order → power leaks through a penalty, or a released DRA still blocks | **One pure `resolveInputs`** with a fixed pipeline: power-inhibitors compose by OR, brake-demands by `max`. Order-independent by construction. | O3–O5, O12 |
| **R3** | **Emergency change breaks existing oracles** | The braking-distance oracle (`physics.test.ts:109`) drives `brake:1` and asserts `brakeServiceDecel`. Reinterpreting `brake=1` as emergency breaks it. | Keep `brake∈[0,1]` scaling **service** decel. Add a **separate `emergency` channel** through `AccelInputs`/`SimInputs`→`brakeForce`, default `false`. | O1–O2, regression |
| **R4** | **Wall-clock / determinism leak** | A `performance.now()`/`setTimeout` in DSD → non-reproducible, frame-rate-dependent vigilance | **All timed safety advances off the explicit `dt`** passed to `tickSafety`. No `Date`/`performance.now`/`setTimeout`/`Math.random` anywhere under `src/sim/`. Verifiable by grep + a determinism oracle. | O8 |
| **R5** | **`brakeActual=1` start mismatch** | `createInitialState` seeds `brakeActual=1`. If the brake controller starts at RELEASE, HUD shows a released lever against a fully-applied brake and frame 1 could roll. | **`createInitialControls()` seeds `brakeStep = FULL SERVICE`** so lever, demand, and `brakeActual` all agree at spawn. | O3 (no-creep) |

**Why one pure brain (`controls.ts`) + thin shell.** The whole control/safety/HUD-derivation logic is a single pure module under `src/sim/`, unit-tested with the same discipline as `physics.ts`. Dependency direction: `main.ts` → {`sim/*`, `ui/hud`}. The `sim/*` modules never import from `ui` or `render`. `controls.ts` imports `safety/aws.ts`. Pure modules import no DOM, no `three`.

### 2.2 Core types

```ts
// src/sim/controls.ts
export type Reverser = "FWD" | "OFF" | "REV";
export type PenaltyReason = "DSD" | "AWS" | "TPWS"; // AWS/TPWS reserved for Phase 2

export const POWER_NOTCHES = 4;                          // OFF + 4 detents
export const BRAKE_SERVICE_STEPS = 3;                    // service steps 1..3
export const BRAKE_FULL = BRAKE_SERVICE_STEPS;           // index 3 = FULL SERVICE
export const BRAKE_EMERGENCY = BRAKE_SERVICE_STEPS + 1;  // index 4 = EMERGENCY (past the end gate)
export const STAND_EPS = 0.3;   // m/s — "at a stand" (reverser, DRA, latch release)
export const DSD_PERIOD = 60;   // s vigilance interval
export const DSD_WARN_WINDOW = 7; // s of warning before penalty

export interface ControlState {
  powerNotch: number;   // 0..POWER_NOTCHES   (integer index; 0 = OFF)
  brakeStep: number;    // 0..BRAKE_EMERGENCY (0=RELEASE, 1..3=service, 3=FULL, 4=EMERGENCY)
  reverser: Reverser;
  lastDir: 1 | -1;      // remembered heading; reverser OFF keeps this, feeds SimInputs.dir; never 0
  dra: boolean;         // Driver's Reminder Appliance: set ⇒ power inhibited
}

export interface SafetyState {
  vigilanceTimer: number;           // s remaining before warning; counts DOWN off dt
  dsdWarning: boolean;              // warning window active (amber lamp)
  penaltyReasons: ReadonlySet<PenaltyReason>; // the SHARED latch: active iff non-empty
}

export function createInitialControls(): ControlState {
  return { powerNotch: 0, brakeStep: BRAKE_FULL, reverser: "OFF", lastDir: 1, dra: false };
  //                                  ^ FULL SERVICE at spawn ⇒ matches brakeActual=1 (R5)
}
export function createInitialSafety(): SafetyState {
  return { vigilanceTimer: DSD_PERIOD, dsdWarning: false, penaltyReasons: new Set() };
}
```

**Integer indices are the source of truth** for power/brake (not floats): notch fraction `powerNotch / POWER_NOTCHES` → exactly {0, .25, .5, .75, 1}; service brake fraction `brakeStep / BRAKE_SERVICE_STEPS` → exactly {0, ⅓, ⅔, 1}. Exact fractions make `notch === 0.5` assertions trivial and avoid float drift. Pure helpers `notchFraction`, `brakeFraction`, `isEmergency`, `brakeApplied`, `penaltyActive(s) = s.penaltyReasons.size > 0` are exported for HUD + tests.

### 2.3 The intent + the inline keymap (in `main.ts`)

`ControlIntent` is a flat record of edges for one frame, applied by the pure `reduceControls`/`tickSafety`:

```ts
// declared in controls.ts (consumed by the pure reducers); produced inline in main.ts
export interface ControlIntent {
  powerUp: boolean; powerDown: boolean;                 // edge: one detent per press
  brakeUp: boolean; brakeDown: boolean;                 // edge
  emergency: boolean;                                   // edge: slam to EMERGENCY
  reverserFwd: boolean; reverserOff: boolean; reverserRev: boolean; // edge
  toggleDra: boolean;                                   // edge
  acknowledge: boolean;                                 // edge: DSD/penalty ack
  vigilancePing: boolean;                               // any control activity this frame ⇒ resets DSD
}
```

`main.ts` maintains a per-frame **edge set** of just-pressed key codes (`keydown` with `if (e.repeat) return` so a hold never auto-repeats a detent; `keyup`/`blur` housekeeping), then builds the intent from a ~20-line literal lookup table (e.g. `KeyW`→powerUp, `KeyS`→powerDown, `KeyD`→brakeUp, `KeyA`→brakeDown, `Backquote`→emergency, `KeyF`/`KeyN`/`KeyR`→reverser FWD/OFF/REV, `KeyL`→toggleDra, `KeyQ`→acknowledge), with `vigilancePing = edgeSet.size > 0`. The table is decision-free data; the only untested logic is the edge-set construction, which is the same handful of lines whether or not the table lives in its own file — so it stays inline (R: simplicity, "don't predict the future").

### 2.4 Control reduction (pure, `controls.ts`)

```ts
export function reduceControls(c: ControlState, intent: ControlIntent, state: SimState): ControlState;
```

Applies, in order, returning a new state (mutates nothing):

1. **Reverser** edges: a change to FWD/OFF/REV is accepted **only if `|state.speed| ≤ STAND_EPS`** (the at-a-stand interlock); otherwise ignored. FWD sets `lastDir=1`; REV sets `lastDir=-1`; OFF keeps `lastDir`.
2. **Power** edges: `powerNotch = clamp(powerNotch ± 1, 0, POWER_NOTCHES)`.
3. **Brake** edges: `brakeUp` steps `0→1→2→3(FULL)→4(EMERGENCY)`; `brakeDown` decrements one step clamped to `[0, BRAKE_EMERGENCY]`. **EMERGENCY is reachable only by stepping past FULL (an end gate)** — a single `brakeUp` from FULL reaches EMERGENCY; it is *not* reachable by single-keypress overshoot from a partial step. The `emergency` edge slams `brakeStep = BRAKE_EMERGENCY` directly.
4. **DRA** toggle: accepted **only at a stand** (`|state.speed| ≤ STAND_EPS`).

`reduceControls` owns the reverser-at-stand and DRA-at-stand guards and the stepped clamps. It does **not** apply the no-power-against-brake interlock — that is a demand-time gate (the lever can sit in notch 2 while effort is suppressed) and belongs in `resolveInputs`.

### 2.5 `resolveInputs` — the interlock pipeline (the PRIMARY SEAM)

The single pure function turning control + safety state into `SimInputs`. Fixed, documented composition order; power-inhibitors OR-compose, brake-demands `max`-compose:

```ts
export function resolveInputs(c: ControlState, s: SafetyState, state: SimState, mu: number): SimInputs;
```

```
1. dir          = c.lastDir                                   // always 1|-1; OFF is expressed as zero notch
2. penaltyOn    = penaltyActive(s)
3. powerInhibit = brakeApplied(c.brakeStep)                   // no power vs applied brake
               || c.reverser === "OFF"                        // reverser OFF inhibits power
               || c.dra                                       // DRA inhibits power
               || penaltyOn                                   // penalty inhibits power
4. notch        = powerInhibit ? 0 : notchFraction(c.powerNotch)
5. emergency    = isEmergency(c.brakeStep)                    // ONLY the driver's EMERGENCY lever; penalty ≠ emergency
6. driverBrake  = brakeFraction(c.brakeStep)                  // EMERGENCY also maps to demand 1
   penaltyBrake = penaltyOn ? 1 : 0                           // penalty = FULL SERVICE demand
   brake        = Math.max(driverBrake, penaltyBrake)
7. return { notch, brake, dir, emergency, mu }
```

Guarantees pinned by oracles: any applied brake OR penalty OR DRA OR reverser-OFF ⇒ `notch === 0`; the driver can never out-rank a penalty (penalty forces full-service brake + zero power regardless of lever positions); **penalty brake is full *service*, not emergency** (`emergency` is reserved for the driver's deliberate EMERGENCY lever — matches the brief's "full-service, latched" penalty).

The safety layer expresses its override purely as `brake`/`notch`/`emergency` floats — exactly the `step()` contract. `step` keeps owning `brakeActual` (the lag); the safety layer never touches it. The *only* new `SimInputs` field is `emergency`, a genuine physics input, not a safety flag.

### 2.6 DSD/vigilance, DRA, and the shared penalty latch (pure, `controls.ts`)

```ts
export function tickSafety(
  s: SafetyState, intent: ControlIntent, state: SimState, dt: number, aws: AwsOutput,
): SafetyState;
```

Order inside `tickSafety` (returns a new state; advances only off `dt` — R4):

1. **Vigilance reset:** if `intent.vigilancePing || intent.acknowledge` → `vigilanceTimer = DSD_PERIOD`, `dsdWarning = false`.
2. **Countdown:** else `vigilanceTimer -= dt`. `dsdWarning = vigilanceTimer ≤ DSD_WARN_WINDOW`. When `vigilanceTimer ≤ 0`, **add `"DSD"`** to `penaltyReasons`.
3. **AWS/TPWS merge:** fold in `aws.reasons` (from the stub, §2.8) at the *same* point DSD adds its reason — Phase 1 contributes none.
4. **Latch release (the one ack path):** apply the per-reason clearability predicate below.

**The CLEAR → WARNING → PENALTY state machine and the release edge:**

```
        ping/ack (timer > WARN)                         ping/ack
        ┌────────────────────────┐                      (timer reset)
        ▼                        │                           ▲
    [CLEAR] ──timer≤WARN──▶ [WARNING] ──timer≤0──▶ [PENALTY: reasons ∋ DSD]
        ▲                        │                           │
        └──── ack edge ──────────┘                           │
                                                             │
   release a reason  ⟺  ack edge  AND  |speed| ≤ STAND_EPS   │
                          AND its source no longer demands it ┘
   latch clears  ⟺  penaltyReasons becomes empty
```

**Documented contract (lives as a comment in `controls.ts`, so the Phase-2 author wiring AWS/TPWS cannot break releasability):**

> A reason is removed from `penaltyReasons` **only when** its source no longer demands it **AND** `|state.speed| ≤ STAND_EPS` **AND** an `acknowledge` edge occurred. The overall latch is "released" only when the set becomes empty. **DSD's source is "expired"** — once the ack-at-stand fires, DSD's demand has lifted and its reason clears (and `vigilanceTimer` resets). Phase-2 AWS reasons that *persist after ack* (magnet still passed) simply fail the "source no longer demands it" clause and slot in **without touching the release machine**. With multiple reasons, one ack at a stand clears all *currently-clearable* reasons; a still-asserting subsystem keeps its reason and the brake stays applied.

**DRA** is **control state**, not a penalty reason: it only inhibits power (§2.5), applies no brake, and is cancelled manually at a stand. It must not feed the latch (a reminder appliance is not a protection event), so it lives in `ControlState` as a boolean + lamp.

### 2.7 The physics emergency-decel change (surgical, R3)

`brakeForce` is called from exactly one site (`acceleration`) and has no direct unit test today, so the blast radius is one parameter. Add an `emergency` flag selecting the decel target:

```ts
// physics.ts
export function brakeForce(
  spec: TrainSpec, brakeActual: number, mu: number, emergency = false,
): number {
  const decel = emergency ? spec.brakeEmergencyDecel : spec.brakeServiceDecel;
  const demanded = clamp01(brakeActual) * decel * effectiveMass(spec);
  const adhesionLimit = mu * spec.mass * G; // braked on all axles
  return Math.max(0, Math.min(demanded, adhesionLimit));
}
```

`AccelInputs` gains a required `emergency: boolean`; `acceleration()` passes `inp.emergency` to `brakeForce` (both the moving branch and the static-hold branch read the same flag). `simulation.ts`: `SimInputs` gains a required `emergency: boolean`; `step()` threads it into the `AccelInputs` it builds. The `brakeActual` lag logic is unchanged — `emergency` selects the *decel scale*; the build-up still rides `brakeActual` toward the `brake=1` demand per `BRAKE_LAG`.

Default `emergency = false` on `brakeForce` keeps the function call API backward-safe, but per the brief's preference for **required fields + defaults over optionals** under `exactOptionalPropertyTypes`, `AccelInputs.emergency` and `SimInputs.emergency` are *required*. The single mechanical ripple is adding `emergency: false` to the literals in `physics.test.ts` (the `AccelInputs` in the RK4 oracle and every `SimInputs` in `drive(...)` call sites) — no assertion weakened. Why a flag and not `brake=1`: `brake∈[0,1] × serviceDecel=0.9` can never reach `1.3 m/s²`; emergency must be a genuinely distinct deceleration with its own closed-form stop distance `v²/(2·1.3)`.

### 2.8 AWS/TPWS future seam (typed no-op stub)

The cheapest shape Phase 2 can extend without rework — a `SafetyDemand`-shaped output and one pure function, homed under `src/sim/safety/`:

import type { PenaltyReason } from "../controls";

### 2.9 The structured HUD

`buildHudView` is **pure and lives in `controls.ts`** (so it is unit-tested without any DOM dependency); `createHud` in `src/ui/hud.ts` is a thin, untested DOM writer:

// buildHudView in controls.ts
export interface HudView {

**The load-bearing split, stated crisply:** `buildHudView` is pure and **is** tested — mph (`|speed|·2.236936`), next-stop lookup (first station ahead of `chainage` in `dir`), brake label from `brakeStep`, demand-vs-actual %, and the lamp booleans. **Only its DOM rendering is the untested adapter** — `createHud` builds per-field `<span>`s once and patches `textContent`/lamp classes (amber for DSD warning + DRA set, red for penalty) on `update`. No jsdom dependency is incurred. The HUD is a sibling DOM overlay, not routed through `SceneHandle`.

### 2.10 What stays in the thin, untested `main.ts`

Only impure glue, no decision logic — explicitly the **deliberately-untested adapter surface**:
- create scene/route/spec/HUD and initial `SimState`/`ControlState`/`SafetyState`/`AwsState`.
- `keydown` (`if (e.repeat) return`)/`keyup`/`blur` listeners; the per-frame edge set; the inline keymap lookup (§2.3).
- the rAF loop and **the only wall-clock:** `dt = min((now - last) / 1000, 0.05)`.
- per-tick orchestration (calls in order, no branching):

```
read edge set → intent = intentFromKeys(edgeSet)            // inline lookup
controls = reduceControls(controls, intent, state)
safety   = tickSafety(safety, intent, state, dt, tickAws(aws, state, dt))
inputs   = resolveInputs(controls, safety, state, ADHESION.wetNight)
state    = step(spec, route, state, inputs, dt)
scene.render(state.chainage)
hud.update(buildHudView(state, controls, safety, route, spec))
clear edge set
```

`mu` stays hardwired `ADHESION.wetNight` (no env var, no flag). Every branch that could be wrong lives in a pure, tested function; `main.ts` contains no control/safety/HUD arithmetic.

## 4. Acceptance criteria / test oracles (the build run's definition of done)

All in the existing pure Vitest idiom (`environment:"node"`, no DOM), reusing the `drive(state, inputs, seconds)` helper. Tests are written **first**; modules are shaped to satisfy them. `npm run check` green, zero TS warnings, is the gate.

**Emergency decel — the new pure physics path**
- **O1 (analytic stop distance).** Strip Davis (`davisA/B/C = 0`); `v0 = 20`; `mu = ADHESION.dry` (so the brake is **not** adhesion-limited — see R3 caveat in §5); `{ notch:0, brake:1, dir:1, emergency:true, mu }`. Drive to rest: `end.speed < 1e-3` and `end.chainage ∈ [0.97, 1.03] × v0²/(2·brakeEmergencyDecel)`.
- **O2 (emergency strictly shorter than service).** Same setup, `emergency:false` (full service) vs `emergency:true`: emergency stop distance `<` service stop distance, ratio ≈ `0.9/1.3` within tolerance. Proves the flag changes physics (guards against "faked as brake=1").
- **Regression.** The existing `brakeServiceDecel` braking-distance oracle (`physics.test.ts:109`) and the RK4 oracle still pass with `emergency:false`.

**Interlocks — `resolveInputs`**
- **O3 (no power against any applied brake).** For every `brakeStep ∈ {1,2,3,EMERGENCY}` with `powerNotch = POWER_NOTCHES`, `resolveInputs(...).notch === 0`; with `brakeStep = 0`, `notch === 1`. Then a driving oracle: from `createInitialControls()` (brake = FULL SERVICE, R5) at rest, drive 10 s flat → `|speed| < 1e-6`, no creep.
- **O4 (reverser OFF inhibits power).** `reverser:"OFF", powerNotch:4, brakeStep:0` ⇒ `notch === 0`; `dir === lastDir` (still `1|-1`, never `0`).
- **O5 (exact fractions).** `powerNotch:2 → notch === 0.5`; `brakeStep:2 → brake === 2/3`; `brakeStep:EMERGENCY → brake === 1 && emergency === true`.

**Reverser & DRA interlocks — `reduceControls`**

## 5. Residual risks

1. **EMERGENCY adhesion caveat (keep — not a bug).** EMERGENCY stays distinct from FULL SERVICE *only while not adhesion-limited*. At `ADHESION.wetNight` (μ=0.2) the adhesion limit is `0.2·160000·9.80665/meff ≈ 1.81 m/s²`, which exceeds 1.3, so emergency is **not** clipped at the signature setting — but at very low grip the limit drops below 1.3 and emergency ≈ service. This is physically correct, not a defect; a future reader must not "fix" it. The analytic oracle (O1/O2) therefore uses `ADHESION.dry` (limit ≈ 2.72 m/s² > 1.3), exactly as the existing braking-distance oracle does, so the `v²/(2·1.3)` check stays valid.
2. **Penalty-release ordering (the load-bearing seam).** "Clear only when source no longer demands it AND at a stand AND acked" is subtle and the one place a bug strands the player or makes the latch un-releasable. Mitigated by O10/O11 and the documented per-reason clearability contract in `controls.ts`; the risk is that Phase-1 under-specifies clearability for reasons that persist after ack — the contract names this explicitly so Phase-2 AWS slots in without touching the release machine.
3. **`exactOptionalPropertyTypes` friction.** Making `emergency` a required field ripples into existing `AccelInputs`/`SimInputs` literals (physics test + `main`). Mitigated: it's a required boolean with an obvious `false`; the impact map lists every literal to touch — small and mechanical.
4. **Untested adapter surface** (inline keymap, edge-set construction, rAF, DOM writer) can hide wiring bugs no oracle catches — e.g. not clearing the edge set per frame → double-stepping. Mitigated by keeping it to a few lines (`if (e.repeat) return` + clear-per-frame is the whole logic) and pushing every decision into pure functions; the residual is intentional and named. Manual smoke-test via `vite dev`.
5. **Shared epsilon / feel tuning.** `STAND_EPS = 0.3` is reused for reverser, DRA, and latch release — coherent but tuning one tunes all; `DSD_PERIOD`/`DSD_WARN_WINDOW` are guesses, and linear `notch/Nmax` gives even effort steps that may feel un-prototypical against a real motor curve (the constant-power taper in `tractiveEffort` softens this). All are single-point named constants; oracles assert behaviour *relative to* the constants, not magic numbers, so tuning won't break tests. None is a blocker.
6. **Phase-2 coupling debt.** The latch's `Set<PenaltyReason>` and the `tickAws` seam are designed for AWS/TPWS but unexercised in Phase 1; O13 is the only guard that the seam stays shaped right. Flagged as the PLN's "on-ramp" — kept minimal, not over-built.
